# Annotation Card Layout Redesign

**GitHub Issues:** #236, #284, #80, #140

## Summary

This design collapses annotation cards from their default expanded state (~120-150px) to a compact header (~28px) showing only essential metadata, expanding on click to reveal full content. This densification (4-5x improvement) solves two race-condition bugs (#236, #284) while enabling high-annotation paragraphs to display 4-5 times more cards without scrolling. The implementation includes refactored card builders, E2E test infrastructure with `expand_card()` helpers, and removal of arbitrary timeouts in favor of deterministic waits.

## Definition of Done

1. **Bug fixes (#236, #284):** Race condition in card positioning setup (highlights-ready event missed on SPA navigation) is resolved; cards render at correct positions on initial load without requiring scroll. Height-cache fix eliminates "solitaire collapse" where cards stack at 0px height after scroll recovery.

2. **Collapsed annotation cards:** Cards default to a compact view showing tag name and essential metadata. Clicking expands to show full content (text preview, comments, tag select dropdown). Reduces per-card height from ~120-150px to ~30px, enabling 4-5x density improvement for high-annotation paragraphs.

3. **E2E card helpers:** `expand_card()` and `collapse_card()` helpers in `annotation_helpers.py`. All card-touching E2E tests updated to expand cards before interacting with inner elements (tag-select, comment-input, etc.).

4. **`e2e cards` runner shortcut:** `uv run grimoire e2e cards` runs the ~9 card-touching E2E tests for rapid iteration during card UI work.

5. **Sleep removal in card-touching tests:** Arbitrary `page.wait_for_timeout()` calls in card-touching tests replaced with state-based waits (`element.wait_for(state="visible")`) and JS condition waits (`page.wait_for_function()`), following patterns established in the test-audit branch.

6. **Complexity management:** All functions in files touched by this work must be at or below `max_complexity_allowed = 15` (complexipy). Specifically: `_build_card_header` (currently 18) and `_build_comments_section` (currently 21) in `cards.py` must be refactored down as part of the collapsed card implementation.

**Out of scope:**
- Card positioning redesign (fisheye/differential scroll rate — future brainstorm after collapsed cards ship)
- NiceGUI User fixture demotion (keep in mind for future, not this design)
- Incremental card add/remove (#140 — separate performance issue)

## Glossary

**Compact header:** The collapsed view of an annotation card (~28px height) showing a coloured dot, tag name, author initials, paragraph reference, comment count badge, expand/collapse chevron, locate button, and delete button. This is the default state before user interaction.

**Detail section:** The expandable portion of an annotation card hidden by default, revealed when the compact header is clicked. Contains the tag select dropdown (if user can annotate), full author name, editable paragraph reference, text preview, and comments section with input field.

**Push-down algorithm:** The JS logic in `positionCards()` that calculates vertical positions for all cards by iterating through visible cards and computing each card's minimum Y position as the previous card's Y plus its height plus a gap (`minY = y + height + minGap`). Ensures cards don't overlap and cascade smoothly when heights change.

**Solitaire collapse:** A bug (#284) where cards hidden by scrolling past them revert to 0px height instead of restoring their cached height, causing all remaining cards to stack at the top of the viewport. Fixed by storing `data-cached-height` and using it in the push-down algorithm for `display: none` elements.

**SPA navigation:** Single-Page Application navigation where page transitions occur without full browser refresh. In PromptGrimoire's annotation page, routing changes trigger the `highlights-ready` event to signal when highlights are injected into the DOM, which card setup handlers must catch.

**Height cache:** An HTML attribute `data-cached-height` storing the last-measured offsetHeight of a card element. Used during the push-down algorithm when a card is hidden (e.g. scrolled off-screen) to preserve its contribution to vertical spacing without rendering the element.

**Fisheye:** A visual technique where UI elements near a focal point expand while elements further away shrink progressively, maintaining visibility of all content while emphasizing the active area. Discussed as a future exploration for card positioning once collapsed cards are stable.

**charOffsetToRect:** A JavaScript function (or browser API method) that maps a character offset in text to its bounding rectangle on the page. Used to position card elements relative to their anchor highlights in the annotation sidebar.

**MutationObserver:** A browser API that watches for changes to the DOM tree (attribute changes, child node additions/removals, text modifications). In this design, MutationObserver triggers re-runs of `positionCards()` when card expand/collapse changes their height.

**Author initials:** Abbreviation of the annotator's name derived by splitting on whitespace and hyphens, taking the first character of each segment, and joining with dots. Example: "Brian Ballsun-Stanton" → "B.B.S.", single name "Ada" → "A.". Shown in compact header and falls back to "A." for anonymous contributors.

## Architecture

Annotation cards in the sidebar transition from a single monolithic builder to a two-part compact/detail structure. Cards default to a collapsed compact header (~28px) and expand on click to show full content (~120-150px).

**Card structure:**

- **Compact header** (always visible): coloured dot + tag display name (static label) + author initials + para\_ref + comment count badge + expand/collapse chevron + locate button + delete button. Author initials derived by splitting the display name on whitespace and hyphens, taking first character of each segment, joining with dots (e.g. "Brian Ballsun-Stanton" → "B.B.S.").
- **Detail section** (hidden by default): tag select dropdown (if `can_annotate`), full author name + editable para\_ref, expandable text preview, comments section with input.

**JS positioning fixes:**

- `annotation-card-sync.js` `setupCardPositioning()` checks `window._highlightsReady` at setup time to catch the race condition where `highlights-ready` fires before the listener is registered (#236).
- `positionCards()` caches card heights in `data-cached-height` so that cards hidden with `display: none` use their last-known height instead of 0, preventing the "solitaire collapse" (#284).

**CSS changes:**

- `.ann-card-positioned` default padding reduced from `8px 12px` to `4px 8px` for compact state.
- New `.ann-card-expanded` class restores `padding: 8px 12px` when detail section is visible.

**E2E test infrastructure:**

- `expand_card(page, card_index)` / `collapse_card(page, card_index)` helpers in `annotation_helpers.py` targeting `data-testid="card-expand-btn"`.
- `@pytest.mark.cards` marker on card-touching tests; `uv run grimoire e2e cards` CLI shortcut.
- Arbitrary `page.wait_for_timeout()` calls replaced with `element.wait_for(state="visible")` and `page.wait_for_function()` following test-audit branch patterns.

## Existing Patterns

Investigation found the current card system in `src/promptgrimoire/pages/annotation/cards.py` (477 lines, 7 functions). Cards are built by `_build_annotation_card()` which calls `_build_card_header()` and `_build_comments_section()` — both exceed the `max_complexity_allowed = 15` threshold (18 and 21 respectively). High complexity comes from inline async callback definitions.

The existing pattern of `_build_*` functions that compose NiceGUI elements within a `with` context is maintained. The refactoring decomposes into `_build_compact_header()` and `_build_detail_section()`, each of which may further extract callback factories to stay under complexity limits.

E2E test helpers follow the established pattern in `tests/e2e/annotation_helpers.py` — module-level functions taking `page` and optional `card_index` parameters, using `data-testid` locators per project convention.

The test-audit branch established the sleep-removal pattern: `element.wait_for(state="visible")` for DOM readiness, `page.wait_for_function()` for JS state conditions. This design adopts that pattern for all card-touching tests.

## Implementation Phases

**Spike-first strategy:** Phase 1 is a spike to validate the collapsed card UX before investing in test infrastructure. If the spike validates, phases 2-4 proceed. If the spike fails, it is reverted and the design revisited.

**Spike failure criteria:** The spike fails if any of: (a) the expand/collapse interaction adds more friction than the density improvement justifies (user judgement call during manual verification), (b) the push-down algorithm produces visible jank on expand/collapse that cannot be resolved with CSS transitions alone, or (c) the compact header cannot convey enough information to be useful without expanding (i.e. users must expand every card to orient themselves, defeating the purpose). A failed spike means reverting to flat cards and exploring alternative density strategies (e.g. fisheye, tabbed grouping, or independent sidebar scroll) rather than iterating on collapsed cards indefinitely.

## Additional Considerations

**Expand/collapse and card positioning interaction:** When a card expands, its height increases from ~28px to ~120-150px. The JS `positionCards()` function re-runs on MutationObserver change and updates the push-down algorithm (`minY = y + height + minGap`). The `transition: top 0.15s` CSS creates smooth repositioning of cards below. The height cache (`data-cached-height`) updates on the next measurement cycle when `offsetHeight > 0`.

**Anonymous sharing:** The compact header shows author initials. Under anonymous sharing mode, `anonymise_author()` already returns "Anonymous" — initials would be "A." which is fine. No special handling needed.

**Viewers (non-annotators):** Compact header shows static tag label (not select dropdown) in both compact and expanded views, matching the current behaviour for view-only users. The expand chevron, locate button, and text preview are available to viewers. Comment input is only shown when `can_annotate` is true.

**Load-bearing sleeps (Phase 4 risk):** Some `page.wait_for_timeout()` calls may guard server-side round-trips (e.g. CRDT commit completing) rather than DOM readiness. These cannot be replaced with `element.wait_for(state="visible")` alone since the element may be visible before the server finishes processing. Strategy: use `page.wait_for_function()` to check client-side state that the server pushes (e.g. a JS flag set after WebSocket acknowledgement). If no such signal exists for a particular operation, the sleep may need to remain with a comment explaining why, or a ready signal may need to be added.

**Height-cache staleness on off-screen collapse (known limitation):** If a card is expanded, scrolled off-screen (hidden), and the user collapses it programmatically while hidden, the cached height would be ~120px but the actual rendered height on return would be ~28px. This creates a temporary gap in the push-down algorithm until `positionCards()` re-measures. In practice this is unlikely — users collapse by clicking, which requires the card to be visible. Noted for awareness; no fix planned unless observed in testing.

**Open questions for fisheye/positioning exploration (post-spike):**

- With compact cards at ~28px, is hiding off-screen cards still necessary? For short documents with few annotations, compact cards might fit without hiding. For long documents with many annotations (e.g. 80 across 50 paragraphs), even 28px × 80 = 2240px of off-screen cards would still push visible cards away from their anchored paragraphs.
- **Independent sidebar scroll:** Decouple sidebar scroll rate from document scroll. Cards always rendered, sidebar just scrolls further. Eliminates hide/show entirely but requires new scroll synchronisation logic.
- **Viewport-anchored rendering (virtual list):** Only *render* cards for paragraphs near the viewport, rather than rendering all and hiding with CSS. More efficient but requires tracking which paragraphs are visible and dynamically creating/destroying card elements.
- **Fisheye sizing:** Cards near the active paragraph expand, cards further away progressively shrink. Combines with independent scroll to keep all cards visible while maintaining alignment for the focal area.
- The height-cache fix (#284) is a prerequisite for any variable-height card approach and solves a real bug regardless of future direction.
