# CLI Typer Migration & Complexity Reduction Design

**GitHub Issue:** #211

## Summary

PromptGrimoire's command-line tooling has grown into a single 2758-line file (`cli.py`) that handles everything from running tests to seeding development data to generating documentation. This design migrates that monolith into a `cli/` package with six domain-focused modules (`testing`, `e2e`, `admin`, `seed`, `export`, `docs`), replacing the hodgepodge of argparse subparsers, raw `sys.argv` checks, and per-command `[project.scripts]` entries with a single unified entry point: `uv run grimoire <subcommand>`.

The migration uses Typer as the argument-parsing framework, which provides auto-generated `--help` output, shell completion, and a composable sub-application model that maps cleanly onto the new module structure. Beyond the structural split, four functions in the existing file exceed the project's cognitive complexity threshold of 15; this design prescribes specific helper-extraction strategies for each, bringing all functions into compliance. The migration is phased so the old `cli.py` remains operational until all six modules are complete, at which point the monolith is deleted and documentation updated.

## Definition of Done

1. **cli.py split into modules** â€” the monolithic 2758-line file is replaced by a `cli/` package with domain-focused modules (testing, e2e, admin/users, seeding, export, docs)
2. **Typer framework** â€” all commands use typer for argument parsing, help text, and shell completion; argparse and raw `sys.argv` usage eliminated
3. **Single entry point** â€” `uv run grimoire <subcommand>` as the unified CLI; old `[project.scripts]` entries removed
4. **Complexity compliance** â€” all functions at or below cognitive complexity 15 (the 4 flagged functions refactored: `_stream_with_progress`, `_retry_parallel_failures`, `_run_fail_fast_workers`, `_run_all_workers`)
5. **CLI tests pass and expand** â€” existing CLI tests (`test_cli_header`, `test_cli_parallel`, `test_manage_users`, `test_make_docs`) continue to pass with updated imports; basic typer invocation tests added for commands not currently covered
6. **No breaking changes to argument names** â€” within each command, the same flags/args work as before; command invocation changes from `uv run <command>` to `uv run grimoire <verb>`

## Glossary

- **Typer**: A Python CLI framework built on Click that uses type annotations to declare arguments and options, and Rich for formatted help output. Replaces argparse in this migration.
- **argparse**: Python's standard-library argument parsing module. Used in the existing `cli.py` for the `manage-users` command group; eliminated by this migration.
- **cognitive complexity**: A code metric (distinct from cyclomatic complexity) that measures how difficult a function is to read by penalising nesting depth and branching. Enforced here at a threshold of 15 per function.
- **complexipy**: The Python tool used to measure and gate cognitive complexity. Run as `complexipy src/promptgrimoire/cli/ --max-complexity-allowed 15`.
- **CliRunner**: Typer's (and Click's) test utility that invokes CLI commands in-process without spawning a subprocess, used in unit tests to verify command registration and help output.
- **`[project.scripts]`**: The `pyproject.toml` table that maps short command names (e.g. `test-all`) to Python entry-point callables. This migration replaces the per-command entries with a single `grimoire` entry.
- **sub-app**: Typer's term for a child `typer.Typer()` instance registered on a parent app via `add_typer()`, producing a named command group (e.g. `grimoire test`, `grimoire admin`).
- **pytest passthrough**: The pattern of forwarding extra command-line arguments (e.g. `-k test_foo -x`) through the CLI wrapper directly to the pytest process, using Typer's `context_settings={"allow_extra_args": True}` and `ctx.args`.
- **asyncio.run() wrapper**: The pattern required because Typer does not support `async def` command functions natively; each async implementation function is wrapped by a synchronous Typer command that calls `asyncio.run(_async_impl())`.
- **`asyncio.as_completed()`**: A Python standard-library coroutine that yields futures as they finish, used in the E2E parallel worker orchestration.
- **xdist**: The pytest-xdist plugin that distributes test execution across multiple processes. Referenced in the E2E test runner design.
- **py-spy**: A sampling profiler for Python processes, integrated into the E2E runner for performance diagnostics.
- **Rich**: A Python library for formatted terminal output (colours, tables, progress bars). Used throughout the codebase for CLI output; Typer also uses it internally for help text rendering.
- **`uv run`**: The command used by the `uv` Python package manager to execute scripts defined in `[project.scripts]`. The project's standard way to invoke CLI tools.
- **state machine**: Used here to describe `_stream_with_progress`'s logic, which transitions through phases (collecting â†’ running â†’ summary) as pytest output lines arrive.
- **fish completion**: Shell tab-completion support for the fish shell, enabled by running `uv run grimoire --install-completion fish` after migration. Provided automatically by Typer's `[all]` extra.

Migrate the monolithic `src/promptgrimoire/cli.py` (2758 lines) to a `cli/` package using typer for argument parsing, with domain-focused modules and a single unified entry point.

### Argument Handling Patterns

**Pytest passthrough commands** (test and e2e groups): Use `context_settings={"allow_extra_args": True, "allow_interspersed_args": False}` on the typer command, then pass `ctx.args` through to pytest. This preserves `uv run grimoire test all -k test_foo -x`.

**Admin subcommands** (argparse â†’ typer): Each existing argparse subcommand (`list`, `show`, `create`, `admin`, `instructor`, `enroll`, `unenroll`, `role`) becomes a `@admin_app.command()`. Arguments and options use `typer.Argument()` and `typer.Option()` with the same names.

**Simple commands** (seed, export, docs): Zero or one positional argument, converted directly.

**Async wrappers**: Typer does not support `async def` commands natively. All async functions keep their existing `asyncio.run()` wrapper pattern â€” sync typer command calls `asyncio.run(_async_impl())`.

### Complexity Reduction

Four functions exceed the cognitive complexity threshold of 15. All are reduced by extracting named helpers, with shared helpers where overlap exists.

**`_stream_with_progress` (36 â†’ ~10) in `testing.py`:**
A 3-phase state machine (collecting â†’ running â†’ summary). Extract `_handle_collecting_phase()` and `_handle_running_phase()` so the parent becomes a simple phase-dispatch loop. Shared with `_stream_plain`: extract `_iter_pytest_output(process, log_file)` generator that yields lines with log writing handled.

**`_run_all_workers` (18 â†’ ~10) and `_run_fail_fast_workers` (19 â†’ ~10) in `e2e.py`:**
Both iterate `asyncio.as_completed()` with exception-to-file mapping. Shared helpers: `_resolve_completed_task(done, task_map)` for awaiting and mapping exceptions to filenames, `_report_worker_progress()` for formatting completion output. Unique to fail-fast: `_cancel_pending_tasks(pending)`.

**`_retry_parallel_failures` (23 â†’ ~10) in `e2e.py`:**
DB cloning/dropping in a retry loop. Extract `_prepare_retry_databases()`, `_classify_retry_results()`, `_cleanup_retry_databases()`.

### Shared Infrastructure

`_shared.py` holds cross-module infrastructure: `_pre_test_db_cleanup()`, `_build_test_header()`, the shared `Rich` `Console()` instance, and pytest output regex constants. Both `testing.py` and `e2e.py` import from `_shared.py`, avoiding sibling-to-sibling coupling.

The `_E2E_SERVER_SCRIPT` string constant (~300 lines) stays in `e2e.py` as-is. It is not a complexity issue â€” it's a string template executed in a subprocess.

`cli_loadtest.py` is out of scope. It remains a separate module.

## Existing Patterns

Investigation found `cli_loadtest.py` as the only existing example of a separate CLI module. It uses raw `sys.argv` inspection (`if "--validate" in sys.argv`). This design does not follow that pattern â€” it adopts typer throughout.

The codebase uses `Rich` console for all CLI output. This design continues that pattern. Typer uses Rich internally for help text, so no conflicts arise.

Existing test files (`test_cli_header.py`, `test_cli_parallel.py`, `test_manage_users.py`, `test_make_docs.py`) import individual functions from `promptgrimoire.cli`. These imports will change to the new module paths.

No existing typer usage in the codebase. This introduces the first typer dependency.
