# JS Unit Tests for Annotation Static JS — Design

**GitHub Issue:** #387

## Summary

Add vitest + happy-dom unit tests for all functions in the three annotation static JS files (`annotation-highlight.js`, `annotation-card-sync.js`, `annotation-copy-protection.js`). Integrate as a new lane in `grimoire test all`. Tests use eval-into-global-scope to load the production files without modifying their global-function structure.

## Definition of Done

1. All public functions in the three annotation JS files have unit tests covering their code paths.
2. Tests run via `npx vitest` and are integrated into `grimoire test all` as a new JS lane.
3. The `walkTextNodes` tests mirror the existing Python `extract_text_from_html` test fixtures exactly (parity contract).
4. Pre-commit hook runs vitest on JS test file changes.
5. Tests pass on CI (Node 25.x).

## Acceptance Criteria

### DoD 1: Function coverage

- **AC1.1** Every public function listed in the complexity table below has at least one test.
- **AC1.2** Functions with CC >= 5 have tests for each distinct code path (branches, early returns, edge cases).
- **AC1.3** The three copies of whitespace-collapsing logic (`walkTextNodes`, `findLocalOffset`, `countCollapsed`) are tested with identical inputs to verify they agree.

### DoD 2: Test lane integration

- **AC2.1** `grimoire test all` runs vitest as a lane before or after the pytest unit lane.
- **AC2.2** `grimoire e2e all` includes the JS lane in its summary.
- **AC2.3** Vitest exit code propagates — a JS test failure fails the overall run.
- **AC2.4** A `grimoire test js` subcommand exists for running JS tests independently.

### DoD 3: Python parity

- **AC3.1** `walkTextNodes` tests use the same HTML inputs as `tests/unit/input_pipeline/test_text_extraction.py::TestExtractTextFromHtml` and assert identical collapsed text output.
- **AC3.2** Block tag, skip tag, and whitespace collapsing *behaviour* is tested to match between JS and Python — via behavioural parity (same HTML inputs produce same outputs), not by inspecting the `SKIP_TAGS`/`BLOCK_TAGS`/`WHITESPACE_RE` constants directly. These are `const` declarations and are not accessible from test code after indirect `eval()` — and testing internal data structures would be testing implementation, not behaviour.

### DoD 4: Pre-commit

- **AC4.1** `.pre-commit-config.yaml` includes a vitest hook that runs on `tests/js/**/*.test.js` changes.

### DoD 5: CI compatibility

- **AC5.1** `package.json` pins vitest and happy-dom versions.
- **AC5.2** `node_modules/` is in `.gitignore`.

## Glossary

| Term | Definition |
|------|-----------|
| **collapsed text** | Text after whitespace runs are collapsed to single spaces and whitespace-only text nodes in block containers are skipped. The canonical form for char offset indexing. |
| **char offset** | Zero-based index into the collapsed text stream. Used to map highlights, selections, and cursor positions between Python and JS. |
| **text node map** | Array of `{node, startChar, endChar}` returned by `walkTextNodes`. Maps each DOM text node to its range in the collapsed char stream. |
| **parity contract** | The requirement that JS `walkTextNodes` and Python `extract_text_from_html` produce identical collapsed text for the same HTML input. |

### Loading global-scope JS into vitest

The annotation JS files declare functions in global scope (`function walkTextNodes(root) { ... }`) for NiceGUI `<script src>` compatibility. They cannot be converted to ES modules.

**Pattern:** `tests/js/setup.js` reads each file with `fs.readFileSync`, then `eval()`s the content. Vitest's `setupFiles` config runs this before every test file.

```js
// tests/js/setup.js
import { readFileSync } from 'node:fs';
import { resolve } from 'node:path';

const STATIC_DIR = resolve(__dirname, '../../src/promptgrimoire/static');

const files = [
  'annotation-highlight.js',     // Must be first (others depend on it)
  'annotation-card-sync.js',
  'annotation-copy-protection.js',
];

for (const file of files) {
  const code = readFileSync(resolve(STATIC_DIR, file), 'utf-8');
  // indirect eval runs in global scope
  (0, eval)(code);
}
```

**Why indirect eval `(0, eval)(code)`:** Direct `eval()` runs in the current scope. The comma operator trick forces global-scope evaluation, which is where the functions need to land.

**`const` visibility caveat:** Variables declared with `const` or `let` in the eval'd code land in the global lexical scope but are NOT attached to `globalThis`. This means `SKIP_TAGS`, `BLOCK_TAGS`, and `WHITESPACE_RE` are not directly accessible from test code. This is intentional — tests should verify behaviour (e.g. "script tags are skipped"), not inspect internal constants.

### Mocking browser APIs

happy-dom provides DOM basics (`document`, `Node`, `Element`, text nodes, `querySelector`, `createElement`, etc.) but lacks some APIs the annotation JS uses:

| API | happy-dom support | Mock strategy |
|-----|-------------------|---------------|
| `CSS.highlights` | No | Map-like mock in `setup.js` |
| `Highlight` constructor | No | Stub constructor in `setup.js` |
| `StaticRange` | No | Stub constructor in `setup.js` |
| `getBoundingClientRect()` | Returns zeros | Per-test mock on specific elements |
| `window.getSelection()` | Partial | Mock for selection tests |
| `ResizeObserver` | No | Stub constructor |
| `MutationObserver` | Likely yes | Use happy-dom's implementation |
| `requestAnimationFrame` | Likely yes | Use happy-dom's or `setTimeout` shim |
| `Quasar.Notify.create` | No | Stub on `globalThis.Quasar` |

**`CSS.highlights` mock (in setup.js):**

```js
globalThis.CSS = globalThis.CSS || {};
globalThis.CSS.highlights = new Map();

globalThis.Highlight = class Highlight {
  constructor(...ranges) {
    this.ranges = ranges;
    this.priority = 0;
  }
};

globalThis.StaticRange = class StaticRange {
  constructor({ startContainer, startOffset, endContainer, endOffset }) {
    this.startContainer = startContainer;
    this.startOffset = startOffset;
    this.endContainer = endContainer;
    this.endOffset = endOffset;
  }
};
```

### DOM fixture builders (helpers.js)

Tests need DOM trees with known structure. A helper builds them from concise descriptors:

```js
/**
 * Build a DOM tree from HTML and return the container element.
 * Uses happy-dom's parser (document.createElement + innerHTML).
 */
export function dom(html) {
  const container = document.createElement('div');
  container.innerHTML = html;
  return container;
}

/**
 * Build a DOM tree and return both the container and the
 * walkTextNodes result for convenience.
 */
export function domWithNodes(html) {
  const container = dom(html);
  const textNodes = walkTextNodes(container);
  return { container, textNodes };
}

/**
 * Mock getBoundingClientRect on an element to return a specific DOMRect.
 * Useful for positionCards tests where happy-dom returns all zeros.
 */
export function mockRect(element, rect) {
  element.getBoundingClientRect = () => new DOMRect(
    rect.x ?? 0, rect.y ?? 0, rect.width ?? 0, rect.height ?? 0
  );
}
```

#### BLOCKER 6: `positionCards` mock approach fundamentally broken

`vi.spyOn(window, 'charOffsetToRect')` does NOT intercept `charOffsetToRect` as declared by indirect `(0, eval)()`. The `eval`'d code creates a function declaration in the global scope, but `vi.spyOn` replaces a property on the `window` object. These are **different binding mechanisms** — `function charOffsetToRect(...)` creates a binding in the global lexical environment, while `window.charOffsetToRect` is a property on the global object. In strict mode or ES module context, `vi.spyOn(window, ...)` may not shadow the function declaration.

This is why the `positionCards` tests fail — the mock is never called, so the function uses happy-dom's real (zero-returning) `getBoundingClientRect`.

**Fix options (pick one):**
1. In `setup.js`, after eval, explicitly assign each function to `globalThis`: `globalThis.charOffsetToRect = charOffsetToRect;` — this makes `vi.spyOn(window, ...)` work.
2. Instead of mocking `charOffsetToRect`, mock `getBoundingClientRect` on the individual text node Range objects using `mockRect` from helpers.
3. Create a thin wrapper in the test that saves/restores the global binding.

Option 1 is simplest. But it needs to be done for ALL functions that tests want to spy on, not just `charOffsetToRect`.
