## Incident Summary

Production PromptGrimoire (NiceGUI 3.9.0, Python 3.14, single-process) exhibits
unbounded RSS growth over time. Two crashes in the telemetry window:

- **Epoch 1** (59caff8f): 12h57m uptime, ended by deploy (#433)
- **Epoch 2** (3e8a4e5a, #433 deployed): 13h22m uptime, SIGABRT at ~4.3 GB RSS
  (2026-03-27 12:12 AEDT / 01:12 UTC)

Bandaid deployed: PR #436 adds memory-threshold graceful restart at 3 GB with
manual reconnect button, exit code 75 for systemd restart.

### Key Observations (facts)

1. **RSS is monotonically non-decreasing within each process lifetime.**
   At 15:47, 26 clients connected, RSS = 1327 MB. At 17:27, 18 clients, RSS = 1404 MB.
   At 19:07, 17 clients, RSS = 1431 MB. Load dropped but RSS never fell.

2. **Dict-level cleanup works.** `ws_reg`, `ws_pres_ws`, `ws_pres_cl` all track
   proportionally with `clients_connected`. When clients drop from 71 to 18, these
   counters drop correspondingly. The cleanup paths in `_handle_client_delete()`,
   `PersistenceManager.evict_workspace()`, and `AnnotationDocumentRegistry.remove()`
   execute correctly at the dict level.

3. **`clients_total` vs `clients_connected` gap is usually small (0-14)** during
   steady state. NiceGUI's `Client.delete()` removes from `Client.instances` after
   a 3-second reconnect timeout. The gap represents clients in the timeout window.

4. **Occasional large gaps during load spikes** (48 at 20:22, 52 at 22:27, 164 at
   01:23 post-crash). Under event loop saturation, the `asyncio.sleep(3.0)` in
   NiceGUI's `delete_content()` competes with page-render tasks. Cleanup is
   delayed when the event loop is busiest.

5. **`asyncio_tasks_total` tracks proportionally with connected clients**, not
   accumulating independently. At 17 clients: 155 tasks (~9/client). At 146
   clients: 768 tasks (~5/client). The ratio varies but is proportional, not
   growing independently.

6. **Post-crash restart at 01:23**: 1330 MB RSS in 11 minutes with reconnection
   storm (188 total clients, 24 still connected). This is 4.8x the cold-start
   baseline (276 MB), suggesting per-client allocations are substantial and the
   reconnection storm (many rapid connect/disconnect cycles) exacerbates retention.

### What the evidence does NOT tell us (inference boundaries)

- **Dict cleanup does not prove object GC.** `_workspace_presence.pop()` removes
  the dict reference to `_RemotePresence`, but if anything else references it
  (reference cycle, asyncio task closure, NiceGUI element tree), the Python
  objects persist in the heap. The counters going down tells us dicts are clean,
  not that memory is freed.

- **RSS growth cannot distinguish leak from fragmentation.** CPython's `pymalloc`
  and glibc `malloc` are known to cause severe RSS inflation in long-running,
  high-churn allocation patterns. 15x RSS growth (276 MB to 4.15 GB) is consistent
  with either a genuine leak, severe arena fragmentation, or both. A controlled
  E2E reproduction with `gc.collect()` is required to distinguish these.

- **The synthetic probe's negative result is production-path-unconfirmed.** It
  tested bare NiceGUI elements without real WebSocket lifecycle. The annotation
  page creates a PageState with 66 fields, CRDT documents (avg 15 KB binary
  state), element trees, and multiple closures. These are absent from the
  synthetic test.

## Cleanup Chain (code audit)

Verified by reading source, paths below are in `src/promptgrimoire/`:

### Disconnect path

1. Browser disconnects WebSocket
2. NiceGUI `Client.handle_disconnect()` starts 3s timer
3. If no reconnect: `Client.delete()` → calls `on_client_delete()` handlers
4. Our `on_client_delete()` (`pages/annotation/broadcast.py:251-300`):
   - `_workspace_presence[workspace_key].pop(client_id)` — removes dict entry
   - If last client: `del _workspace_presence[workspace_key]`
   - Persists CRDT state: `pm.force_persist_workspace(workspace_id)`
   - If last client: `pm.evict_workspace()` + `_workspace_registry.remove(doc_id)`

### What cleanup does NOT do

- **Does not null closures.** `handle_update_from_other()` and `handle_peer_left()`
  closures capture `state` (PageState). After `_workspace_presence.pop()`, if
  reference cycles exist, GC must collect them. No explicit nulling occurs.

- **Does not clear `_broadcast_callback`.** `AnnotationDocument.set_broadcast_callback(None)`
  is never called from application code. The callback is only released when the
  AnnotationDocument itself is GC'd (after `_workspace_registry.remove(doc_id)`
  drops the last registry reference). This is a secondary concern — the callback
  only leaks if the document leaks.

- **Does not force `gc.collect()`.** After cleanup, Python's generational GC
  handles reference cycles on its own schedule.

### PageState size (potential retention concern)

`PageState` (`pages/annotation/__init__.py:207-358`) is a large dataclass:
- 66 annotated fields including `crdt_doc` (AnnotationDocument with pycrdt Doc),
  multiple `ui.element` references, `document_chars` (list of chars from HTML),
  `card_snapshots` dict, `document_tabs` dict, closures (`refresh_annotations`,
  `broadcast_update`, etc.)
- If a single reference to PageState survives, the entire object graph is pinned.

### Closure reference chains (speculative)

```
handle_update_from_other (closure) → captures `state` (PageState)
    PageState.broadcast_update → captures `_workspace_presence` (global dict)
    PageState.crdt_doc → AnnotationDocument → _broadcast_callback (never nulled)
    PageState.annotations_container → NiceGUI element tree
```

If PageState → callback → closure → state forms a cycle, CPython's cycle collector
should handle it, but large object graphs can survive multiple GC generations.
This is **speculative** — needs measurement.

## Hypotheses

All hypotheses are **speculative** (evidence grade: very low). No positive or
negative border has been tested on the production code path.

| # | Hypothesis | Mechanism | Why plausible |
|---|-----------|-----------|---------------|
| H1 | Closure reference cycles pin PageState after disconnect | `handle_update_from_other` → `state` → `broadcast_update` → `_workspace_presence` | Code shows closures capture state; no explicit nulling on cleanup |
| H2 | NiceGUI element trees retained after `Client.delete()` | `remove_all_elements()` may not break all internal refs | NiceGUI #4521 (circular refs), #2589 (`_handle_delete` issues) — upstream, unresolved |
| H3 | pycrdt `Doc` objects retain C-level memory | Yrs FFI allocations not returned to Python allocator | CRDT docs avg 15 KB, 130+ loaded concurrently |
| H4 | PersistenceManager cancelled tasks retain closure refs | `task.cancel()` doesn't release closure captures | Cancelled tasks are Python objects until GC'd |
| H5 | Event-loop saturation delays cleanup | Under load, NiceGUI's `asyncio.sleep(3)` → `Client.delete()` can't fire promptly | Post-crash: 164 gap at 01:23; cleanup competes with renders |
| H6 | malloc arena fragmentation (not a leak) | CPython/glibc don't return freed pages to OS | Known behaviour in long-running Python; 15x growth is plausible for fragmentation alone |

**H6 is the null hypothesis.** If the E2E probe shows RSS stabilises after
`gc.collect()` + `malloc_trim()`, the growth is fragmentation, not a leak.

## Phase 3: Memory Probe Results

### Single-user probe (nicegui_user fixture)

Test: `tests/integration/test_memory_leak_probe.py::TestMemoryLeakProbe::test_memory_growth_across_cycles`

Uses `nicegui_user` fixture with the production-extracted heavy workspace
(`workspace_pabai_190hl_scrubbed.json`): 181 KB CRDT state, 190 highlights,
11 tags, 426 KB document. 10 connect/disconnect cycles, each cycle:

1. `nicegui_user.open("/annotation?workspace_id=<id>")` — full page render
2. Wait for `doc-container` test ID (confirms CRDT load + element tree build)
3. Explicit `Client.delete()` on old clients (exercises cleanup chain)
4. `nicegui_user.open("/login")` — navigate away

6. `gc.collect()` x 2, measure again
7. `malloc_trim()`, measure again

### Predictions (stated before run)

- If RSS grows linearly per cycle after `gc.collect()`: genuine leak (H1-H5)
- If RSS stabilises after `gc.collect()`: fragmentation (H6), no code fix needed

### Analysis

**Important caveats on this probe's cleanup path:** The `nicegui_user`
fixture does NOT trigger the real WebSocket disconnect → `Client.delete()`
path. The probe manually calls `Client.delete()` on old clients
(test_memory_leak_probe.py:368-372) to exercise the cleanup chain.
This means the probe tests whether cleanup **can** free memory, not
whether production cleanup **does** free memory. Results below are
conditional on forced cleanup + forced GC.

1. **Under forced cleanup + GC, RSS is flat.** RSS stabilises at 385 MB
   after `gc.collect()` + `malloc_trim()` across cycles 2-10. The
   prediction for H6 (fragmentation) is matched for this cleanup path.

2. **Dict-level counters reach zero; deeper object retention not measured.**
   `Client.instances`, `_workspace_registry`, `_workspace_presence`,
   `asyncio.all_tasks()` are the only counters checked. PageState instances,
   NiceGUI element trees, pycrdt Doc objects, and closures are **not
   individually counted**. The flat RSS suggests they are freed, but this
   is inference from aggregate RSS, not direct measurement.

3. **H1-H5 weakened but not ruled out.** With explicit `Client.delete()`
   and `gc.collect()`, the ~168K objects per cycle are collected and RSS
   does not grow. This weakens H1-H4 (the objects are **collectible**)
   but does not rule them out under production conditions where cleanup
   is natural, not forced, and GC runs on its own schedule.

4. **H5 (event-loop saturation delays cleanup) remains plausible as an
   amplifier.** The probe runs single-user with no event-loop contention.
   In production under load, `Client.delete()` may be delayed, and
   `gc.collect()` never runs explicitly. Collectible objects may persist
   longer than expected under load.

**H6 is plausible but not yet the confirmed explanation.** The single-user

cleanup. But this probe bypasses the real WebSocket disconnect path and
forces cleanup that production does not perform. See E2E probe below.

### Limitations (epistemic boundary)

- **Single-user test.** Does not exercise concurrent access, WebSocket
  multiplexing, or event-loop contention. H5 cannot be tested here.
- **nicegui_user fixture vs real WebSocket.** The `nicegui_user` fixture
  simulates HTTP-level client interaction, not real WebSocket frames. Some
  NiceGUI internal paths may differ.
- **No interaction beyond page load.** The probe loads the page and
  disconnects. It does not exercise CRDT edits, tag operations, or
  highlight creation within a session.
- **Evidence grade: plausible, not demonstrated.** To reach "demonstrated"
  for H6, we would need to show the same flat-after-GC pattern under
  concurrent multi-user load, AND show that removing GC/trim causes growth.
  The negative border (RSS grows without GC) is inferred from production
  telemetry, not directly tested.

### E2E probe: 10 concurrent Playwright browsers (2026-03-28)

Test: `tests/e2e/test_memory_probe_434.py::TestMemoryProbe434::test_concurrent_memory_growth`

Real Playwright browsers (Chromium), 10 contexts per cycle, 5 cycles. Same
Pabai workspace (190 highlights, 181 KB CRDT state, 426 KB document content,
761 KB fixture file total). Each cycle: authenticate 10 users → all open
annotation page → wait for full render → close all browser contexts →
wait 3s for WebSocket disconnect → **`/api/test/cleanup`** (force-deletes
stale NiceGUI clients, disconnects orphan engine.io sessions, cancels
orphan `Event.wait` tasks) → `/api/test/gc` (gc.collect x 2 + malloc_trim)
→ measure via `/api/test/diagnostics`.

**Cleanup caveat:** The `/api/test/cleanup` endpoint force-deletes clients
and cancels orphan tasks that production relies on natural timeouts to clear.
This means the probe measures retention **after aggressive forced cleanup**,
not after production-like natural cleanup. The +9 MB/cycle residue is a
lower bound — production retention per cycle may be higher.

Full output: `output/incident/e2e_probe_results.txt`

| Metric | Baseline | Cycle 1 | Cycle 5 | Trend |
|--------|----------|---------|---------|-------|
| RSS after gc+trim (MB) | 283 | 486 | 531 | **+9 MB/cycle** |
| RSS with 10 connected (MB) | — | 574 | 598 | +6 MB/cycle |
| gc_collected | 256 | 726,567 | 756,696 | ~Constant |
| clients after GC | 0 | 0 | 0 | Clean |
| presence_clients after GC | 0 | 0 | 0 | Clean |
| ws_registry after GC | 0 | 0 | 0 | Clean |
| asyncio_tasks | 12 | 13 | 13 | Constant |

#### Analysis

1. **Small but measurable RSS growth: ~9 MB per cycle of 10 clients.**
   RSS after gc+trim: 486 → 506 → 510 → 517 → 531 MB. Growth ratio
   over 5 cycles: 1.09x. Well under the 1.5x alarm threshold, but
   **not flat** like the single-user NiceGUI probe showed.

2. **Five application-level counters reach zero after forced cleanup.**
   `nicegui_clients`, `presence_total_clients`, `ws_registry`,
   `asyncio_tasks` all drop to 0/baseline after `/api/test/cleanup` + GC.
   However, **these are the only counters measured**. PageState instances,
   NiceGUI element trees, pycrdt Doc objects, closure references, and
   socketio session data are not individually tracked. The retained
   ~9 MB/cycle may include un-measured Python objects, not just
   allocator fragmentation.

3. **~0.9 MB per client retained after forced GC + malloc_trim.** This is
   memory that survives aggressive forced cleanup — candidates:
   - glibc arena internal fragmentation (small allocations between large
     ones prevent page release even after malloc_trim; `malloc_trim(0)`
     cannot reclaim partially-filled thread-local arenas)
   - pycrdt Yrs FFI allocations (Rust-side memory managed by a separate
     allocator invisible to Python GC and glibc malloc_trim)
   - NiceGUI internal structures not covered by `Client.delete()`
     (known circular reference issue #5110 in element slot children)
   - python-socketio per-session state (request objects, event handlers)

   (RSS is measured server-side via `/proc/self/status`, so browser-side
   memory is not a factor.)

4. **Production extrapolation (speculative).** At ~0.9 MB/client retained
   under forced cleanup, production retention is likely higher because:
   - Production has no forced cleanup or explicit GC
   - Event loop saturation delays natural cleanup (H5)
   - The probe does not exercise CRDT edits, tag operations, or highlight
     creation — only page load and disconnect
   A rough estimate of 200 clients/day x ≥0.9 MB ≈ ≥180 MB/day is
   plausible but imprecise.

5. **The #436 memory-threshold restart is the correct operational
   mitigation.** Whether the residue is allocator fragmentation,
   un-measured Python object retention, or both, the threshold restart
   at 3 GB keeps the process healthy. Further investigation to pinpoint
   the exact allocator would require C-level profiling (jemalloc swap,
   `MALLOC_ARENA_MAX` tuning, `/proc/[pid]/smaps` analysis) — see
   Recommendations below.

### Natural cleanup probe (2026-03-28)

Test: `test_memory_probe_434.py::TestMemoryProbe434::test_natural_cleanup_memory_growth`

Same as the forced-cleanup E2E probe but **without `/api/test/cleanup`**.

#### Analysis

1. **Natural cleanup works.** All 5 tracked counters (clients, presence,
   registry, tasks) reach zero without forced cleanup. NiceGUI's
   reconnect_timeout → `Client.delete()` → `on_client_delete()` chain fires
   correctly for all 10 clients within 5s.

2. **Natural cleanup retains ~6 MB/cycle vs forced cleanup's ~1 MB/cycle.**
   The ~5 MB/cycle difference (~0.5 MB/client) represents memory that
   `/api/test/cleanup`'s extra steps reclaim. That endpoint does three

   engine.io sessions, and cancel orphan `Event.wait` tasks. Which of
   these three accounts for the gap is not yet isolated.

3. **Both variants show the same pattern:** tracked counters reach zero,
   but RSS grows. The retained memory is below the 5-counter measurement
   surface in both cases.

4. **The natural cleanup result partially validates H5.** Even with a fast
   0.5s reconnect_timeout (vs 30s in production), natural cleanup leaves
   ~5x more residue per cycle than forced cleanup. In production with 30s
   timeout and event-loop contention, the gap is likely larger.

#### Updated evidence grading

| # | Hypothesis | Grade | Rationale |
|---|-----------|-------|-----------|
| H1-H4 | Application-level leaks | **Weakened** — possible | Tracked counters zero under both forced and natural cleanup. Deeper objects not measured. Objects are collectible but ~0.5 MB/client retained under natural conditions that forced cleanup can reclaim |
| H5 | Cleanup path leaves residue | **Plausible** (strengthened) | Natural cleanup retains ~6 MB/cycle vs forced ~1 MB/cycle. The ~5 MB gap is memory that natural cleanup doesn't reach. Production (30s timeout, event-loop load) likely retains more |
| H6 | glibc arena fragmentation | **Plausible** | ~1 MB/cycle residue survives even forced cleanup + GC + trim. Consistent with arena fragmentation that no application-level action can reclaim |
| H7 | Sub-allocator retention | **Possible** | The ~1 MB/cycle forced-cleanup residue could be pycrdt Yrs FFI, NiceGUI #5110 circular refs, or glibc internal fragmentation. Cannot distinguish without allocator-level profiling |

#### Epistemic boundary

- **Plausible:** Natural cleanup leaves ~6 MB/cycle residue; forced cleanup
  leaves ~1 MB/cycle. The ~5 MB gap is attributable to the extra teardown
  in `/api/test/cleanup` (H5) but which of its three actions matters is
  not isolated. The remaining ~1 MB/cycle survives even aggressive forced
  cleanup (H6/H7).
- **Possible:** The ~5 MB gap could also include un-measured Python objects
  (PageState, element trees, closures) that forced cleanup's extra teardown
  steps happen to break references for. NiceGUI #5110 (circular refs in
  slot children) is a known upstream issue.
- **Not tested:** Which sub-allocator holds the ~1 MB/cycle forced-cleanup
  residue. Would require `MALLOC_ARENA_MAX` tuning, jemalloc swap, or
  `/proc/[pid]/smaps` analysis.
- **Practical conclusion:** The #436 threshold restart is the correct
  operational mitigation. Natural cleanup works (counters reach zero) but
  leaves ~0.6 MB/client/cycle that accumulates over a process lifetime.
  At 200 clients/day, that's ~120 MB/day minimum — hitting the 3 GB
  threshold in ~23 days under ideal conditions, faster under load.

## Recommendations

1. **Keep #436 threshold restart** as the primary operational mitigation.

2. **Test `MALLOC_ARENA_MAX=2`** in the systemd unit as a tuning hypothesis.
   Constrains glibc to 2 arenas instead of 32, which external guides claim
   reduces fragmentation at the cost of increased lock contention. Claimed
   tradeoffs have not been measured locally and may differ under our workload.
   Requires load testing before production deployment.

3. **Optional: add deeper object counts to diagnostics endpoint.** Track
   PageState instances, pycrdt Doc count, NiceGUI element tree size per
   client. Would narrow whether the ~5 MB/cycle natural-cleanup gap is
   Python objects (fixable) or allocator fragmentation (mitigate with
   arena tuning).

4. **In progress: isolate which `/api/test/cleanup` actions reclaim the
   ~5 MB gap.** The cleanup endpoint does three things beyond natural
   cleanup: force-delete NiceGUI clients, disconnect orphan engine.io
   sessions, and cancel orphan `Event.wait` tasks. The evidence only
   supports "the combined extra teardown reclaims ~5 MB/cycle" — it does
   not yet identify which action(s) matter. Three probe variants
   (`test_cleanup_gap_{clients,eio,events}_only`) have been added to
   isolate each action individually. Results will identify whether a
   specific fix for `on_client_delete()` is warranted and what it
   should target.

## Phase 4: Cleanup Gap Isolation Experiment

### Motivation

Prior phases established that natural cleanup retains ~6 MB/cycle while
forced cleanup retains ~1 MB/cycle. The ~5 MB/cycle gap is attributable
to the combined extra teardown in `/api/test/cleanup`, but we do not
know which of its three actions matters. Without isolation, any fix
would be guessing.

### Experimental design

**Independent variable:** Which single cleanup action runs after natural
cleanup completes.

**Dependent variable:** Per-cycle RSS growth (MB) after GC + malloc_trim.

**Controls:**
- Same workspace fixture (Pabai: 190 highlights, 181 KB CRDT, 426 KB doc)
- Same cycle count (5 cycles x 10 Playwright browsers)
- Same E2E server with 0.5s reconnect_timeout
- Natural cleanup runs first in all variants (5s wait for NiceGUI's
  Client.delete() → on_client_delete() chain to complete). The 5s budget
  covers: 0.5s reconnect_timeout + 3.0s `delete_content()` sleep + ~1.5s
  margin for 10 concurrent clients' event-loop scheduling. Phase 3 showed
  all 5 counters reach zero within this window on a clean server.
  `presence_clients` is used as the sentinel because it is the last
  counter zeroed by the cleanup chain (after dict removal, CRDT persist,
  and workspace eviction). If `presence_clients` is nonzero after the
  5s wait, natural cleanup did not complete — each variant asserts this
  on an `after_natural` snapshot taken before the targeted action runs.
- Each variant then runs exactly ONE targeted cleanup action via
  `/api/test/cleanup?mode=X`

**Three variants:**

| Variant | Mode | What it does after natural cleanup |
|---------|------|------------------------------------|
| clients_only | `clients_only` | Force-delete any remaining NiceGUI Client instances + their socket.io SIDs |
| eio_only | `eio_only` | Disconnect orphan engine.io sessions (WebSocket receive tasks surviving after client deletion) |
| events_only | `events_only` | Cancel orphan `Event.wait` tasks (leaked by NiceGUI's page handler) |

**Reference baselines (from Phase 3, fresh server):**
- Natural cleanup (no extra action): ~6 MB/cycle
- Forced cleanup (all three actions): ~1 MB/cycle

These baselines were measured on a freshly started server. The isolation
variants run on a warm server after prior tests, so absolute growth
rates may differ from Phase 3. **The primary comparison is
variant-to-variant within the isolation class**, not against Phase 3
absolute values. Phase 3 ranges are rough references only.

### Predictions (stated before running)

The predictions below use Phase 3 ranges as approximate anchors. The
diagnostic signal is which variant(s) show *substantially lower* growth
than the others, not whether any variant exactly matches Phase 3 values.

1. **If orphan engine.io sessions account for the gap:** `eio_only` will
   show substantially lower per-cycle growth than the other two variants.
   This would mean python-socketio per-session objects are the primary
   retention source. Fix: add engine.io session sweep to
   `on_client_delete()`.

2. **If Event.wait task accumulation accounts for the gap:** `events_only`
   will show substantially lower per-cycle growth than the other two.
   This would mean NiceGUI's page handler leak (documented in
   `docs/e2e-debugging.md`) is the primary retention source. Fix: cancel
   orphan Event.wait tasks in `on_client_delete()` or upstream. The
   `events_only` test asserts `orphan_wait > 0` to confirm the
   `Event.wait` qualname filter is actually matching tasks — a zero
   match would silently invalidate the variant.

3. **If stale NiceGUI clients account for the gap:** `clients_only` will
   show substantially lower per-cycle growth. This would be surprising
   since natural cleanup already zeros `Client.instances` — it would
   suggest the counter reaches zero but some client state survives.

4. **If multiple actions contribute:** Two or more variants will show
   intermediate growth (lower than natural but higher than forced).
   Individual contributions may not sum cleanly to ~5 MB due to ordering
   and interaction effects between actions.

5. **Null result (falsification):** If all three variants show growth
   comparable to each other and to natural cleanup (~6 MB/cycle), then
   either (a) the actions require each other (ordering dependency),
   (b) the gap is measurement noise amplified by execution order in the
   Phase 3 comparison, or (c) the gap is caused by something the cleanup
   endpoint does implicitly (e.g. `asyncio.sleep(0)` giving the event
   loop time to process pending callbacks).

### Execution

Run: `uv run grimoire e2e perf -k test_cleanup_gap`

Each variant runs in the same pytest session, sharing the same server
process. This means later variants inherit residue from earlier ones.
The per-cycle *growth rate* (not absolute RSS) is the metric of interest.

**Confound: cumulative server state.** If an early variant produces high
residue, later variants start from a higher RSS baseline. This could
cause growth-rate compression if arena fragmentation saturates. The
direction of bias is: later variants may appear to grow slower simply
because the allocator has fewer reclaimable pages, not because their
cleanup action is more effective. Pytest collects these tests in
alphabetical order: `clients_only` → `eio_only` → `events_only`.

**Required confirmation step.** The shared-session run identifies
candidate winner(s) but does not confirm them. Any variant that shows
substantially lower growth must be re-run in an isolated server session
to rule out saturation effects before trusting the result:
```bash
uv run grimoire e2e perf -k test_cleanup_gap_eio_only
```
If the isolated run reproduces the lower growth rate, the result is
confirmed. If not, the shared-session result was an order artefact.

**Contamination guard.** Each isolation variant takes a diagnostic
snapshot after the 5s natural wait but before the targeted cleanup
action runs (the `after_natural` phase in output). If `presence_clients`
is nonzero on any `after_natural` snapshot, the natural cleanup was
incomplete and the targeted action may have masked it — the test fails
with a contamination message.

**Positive controls.** Each variant asserts its targeted action matched
at least one target (e.g. `eio_closed > 0` for `eio_only`). A zero
match fails the test as "ambiguous." Note: for `clients_only` and
`eio_only`, a zero match is itself informative — it would mean natural
cleanup leaves no such targets to reclaim, which narrows the gap's
cause. If a positive control fails, check the cleanup response counts
before dismissing the variant.

Output files: `output/incident/e2e_probe_{clients,eio,events}_only.txt`
