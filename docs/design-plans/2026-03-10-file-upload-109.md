# File Upload Support: DOCX & PDF Import

**GitHub Issue:** #109 (also addresses #238)

## Summary

PromptGrimoire currently accepts legal documents only as pasted HTML from a browser. This feature adds first-class DOCX and PDF upload support so that instructors and students can upload files directly — a DOCX is converted to semantic HTML via mammoth (preserving headings, bold/italic, lists, and paragraphs), while a PDF is converted to structured Markdown via pymupdf4llm and then to HTML via pandoc. Both paths feed into the existing input pipeline and produce annotatable content with no changes to the annotation or export subsystems downstream.

Beyond conversion, the feature makes two structural improvements to the annotation page. A lightweight "edit mode" using NiceGUI's built-in WYSIWYG editor lets users correct conversion artefacts (such as flat-text table headers from AustLII metadata) on documents that have not yet been annotated, writing changes directly to the database rather than going through the CRDT collaboration layer. The page-reload that currently occurs after adding a document is also eliminated by wrapping the document rendering function in NiceGUI's `@ui.refreshable` decorator, so that only the document container rebuilds after an upload or edit. A parallel fix (issue #238) adds explicit unit-test coverage for paragraph preservation when content is pasted from a PDF viewer rather than uploaded.

## Definition of Done

1. **DOCX upload** works end-to-end: upload → HTML conversion preserving paragraphs, headings, block quotes, numbered lists → annotation pipeline → export PDF. Fixture: *Shen v R* [2024] NSWCCA 252 (`2025 LAWS1000 case.docx`) renders correctly.
2. **PDF upload** works end-to-end: upload → text extraction with paragraph structure → annotation pipeline → export PDF.
3. **Edit mode** exists: when a document has zero annotations, user can open `ui.editor` WYSIWYG to fix content, then save back. Not CRDT-backed — this is a "fix it" mode, not a collaboration mode.
4. **No page reload** after adding or editing a document — the new/edited content appears in-place.
5. **Complexipy compliance**: all touched files refactored to stay under threshold 15, reduced total lines.
6. **#238 unit test**: newline/paragraph preservation for PDF-pasted content has an explicit unit test, not just incidental coverage.

**Out of scope:** RTF, OCR for scanned PDFs, multi-document switching UI (#186 beyond reload removal), drag-and-drop.

## Glossary

- **annotation pipeline**: The processing chain that takes a document's HTML content and makes it available for highlighting, commenting, and tagging by multiple users.
- **AustLII**: Australasian Legal Information Institute — the primary free online source for Australian case law. Documents in this project are often copied or printed from AustLII.
- **CRDT**: Conflict-free Replicated Data Type. The annotation page uses pycrdt to synchronise highlights and comments across users. Edit mode deliberately bypasses CRDT because it is a single-user "fix" operation.
- **complexipy**: Static analysis tool measuring cognitive complexity of Python functions. This project enforces a maximum of 15 per function.
- **content_form.py**: The 1,042-line NiceGUI component handling document input (paste, upload, edit). Primary refactoring target in Phase 5.
- **`@ui.refreshable`**: NiceGUI decorator that marks a rendering function so it can be re-run in-place without rebuilding the whole page. Accepts parameters on `.refresh()` — new kwargs merge into stored kwargs (`refreshable.py:127`).
- **`ui.state()`**: React-style state variable inside a `@ui.refreshable`. Returns `(value, setter)`. Calling the setter triggers an automatic refresh of the containing refreshable. State persists across refreshes but resets when the refreshable is called with new parameters.
- **mammoth**: Python library converting DOCX (Word XML) directly to semantic HTML, preserving heading levels, bold, italic, lists, and paragraphs.
- **magic bytes**: First bytes of a binary file identifying its format (e.g., DOCX starts with `PK\x03\x04`). Used by content-type detection in `html_input.py`.
- **NBSP (`\xa0`)**: Non-breaking space character. Originally hypothesised as a PDF paste paragraph separator (AC6.1), but real-world investigation confirmed zero NBSP characters in PDF paste from Chrome and Evince. NBSP runs appear only for in-table LI spacing. AC6.1 dropped.
- **NotImplementedError seam**: Deliberate placeholder at `html_input.py:849` marking where new converters are inserted.
- **pandoc**: Universal document converter. Already used in the export pipeline for HTML→LaTeX; reused here for Markdown→HTML.
- **pymupdf4llm**: Extension to PyMuPDF that extracts structured Markdown from PDFs using AI-based layout analysis, preserving paragraph and list structure.
- **QEditor / `ui.editor`**: NiceGUI's wrapper around Quasar QEditor — a WYSIWYG rich-text editor. Used for edit mode.
- **WorkspaceDocument**: SQLModel table storing a document's HTML content and paragraph map within a workspace.

## Architecture

Two converters plug into the existing `process_input()` pipeline at the `NotImplementedError` seam (`input_pipeline/html_input.py:849`). Both produce HTML strings that feed into the existing preprocessing chain (strip attributes, remove empties).

**DOCX path:** mammoth converts Word XML directly to semantic HTML. Headings, lists, bold/italic, and paragraph structure are preserved. No intermediate format.

**PDF path:** pymupdf4llm (with pymupdf.layout for AI-based layout analysis) extracts structured Markdown. pandoc converts the Markdown to HTML. This two-step approach leverages existing dependencies — pymupdf is already in pyproject.toml, pandoc is installed system-wide and used in `export/pandoc.py`.

**Edit mode** uses NiceGUI's `ui.editor` (Quasar QEditor WYSIWYG) in a wide dialog (80vw, 60vh) launched from the Manage Documents dialog. A pure guard function `can_edit_document()` checks both zero annotations AND non-template-clone status before showing the edit button. Not CRDT-backed — the edited HTML writes directly to `WorkspaceDocument.content` via `update_document_content()`, which also rebuilds the `paragraph_map` and sets `search_dirty=True`.

**Reload removal** uses `@ui.refreshable` with parameters on the document rendering function. The refreshable accepts `doc_id: UUID` as a parameter; switching documents calls `.refresh(doc_id=new_id)`. NiceGUI's `RefreshableTarget` stores args/kwargs per client and merges updates on refresh (see `refreshable.py:126-127`). Edit mode uses `ui.state(False)` inside the refreshable — the setter triggers auto-refresh, and the state resets when switching documents. This is the established pattern used in `courses.py:1261`, `sharing.py:138`, `header.py:224`, extended with parameterised refresh for future multi-document support (#186).

### Data Flow

```
Upload/Paste → detect_content_type()
                    ↓
            ┌───────┴───────┐
            │               │
        "docx"           "pdf"
            │               │
     mammoth →         pymupdf4llm →
     HTML              Markdown → pandoc → HTML
            │               │
            └───────┬───────┘
                    ↓
          process_input() Steps 2-4
          (preprocess, strip attrs, remove empties)
                    ↓
          add_document() / update_document_content()
                    ↓
          @ui.refreshable → re-render document container
```

## Existing Patterns

**`@ui.refreshable`** — Used in 5 places across the codebase. The pattern in `sharing.py:138` (rebuild shares list after permission change) is the closest analogue to rebuilding the document container after add/edit.

**Async subprocess for pandoc** — `export/pandoc.py:270-293` uses `asyncio.create_subprocess_exec()` with stdout/stderr pipes and returncode checking. The PDF converter will follow this pattern with `-f markdown -t html` instead of `-f html+native_divs -t latex`.

**Content type detection** — `html_input.py:123-147` already detects DOCX and PDF via magic bytes and returns the appropriate `ContentType` literal. The upload handler in `content_form.py:119-121` already routes on these types — the `NotImplementedError` is the only gap.

**Document CRUD** — `db/workspace_documents.py` has `add_document()` (line 21), `update_document_paragraph_settings()` (line 124), and `delete_document()` (line 149). The new `update_document_content()` fits between the existing update and delete functions.

**No existing "zero annotations" guard** — `document_management.py` guards on template clones (line 119) but not on annotation count. Edit mode introduces a new guard that queries the CRDT for highlights on the document_id.

**Architectural deviation from original design:** The edit button and editor live in `document_management.py` (the Manage Documents dialog), not in `document.py`'s `@ui.refreshable`. Instead of toggling an in-place `ui.state(editing)` flag inside the refreshable, the implementation closes the narrow management dialog and opens a separate wide dialog with `ui.editor`. This avoids coupling the editor to the refreshable lifecycle and keeps the document view purely a read-only renderer.

Template clones are excluded because editing would desynchronise them from the source template, and the char offsets would no longer match any annotations that exist on the source.

## Additional Considerations

**Converter output quality:** mammoth and pymupdf4llm produce "good enough" output for annotation. Borderless tables (e.g., AustLII case metadata headers) render as flat text in both converters — neither tool detects positional table layouts. Edit mode is the escape hatch for fixing conversion artefacts. When students paste directly from AustLII in a browser, the HTML table structure is preserved by the existing pipeline.

**Edit mode and live sessions:** When `update_document_content()` writes new HTML, other clients holding CRDT sessions will have stale paragraph maps. The zero-annotations guard is the primary defence — no annotations means no offset-dependent state to corrupt. If other clients have the workspace open (e.g., instructor previewing), they should be notified to reload the document. Implementation should set `search_dirty=True` and consider broadcasting a document-changed event via the existing CRDT sync channel.

**Test fixtures:** Two real legal documents serve as fixtures:
- `2025 LAWS1000 case.docx` — *Shen v R* [2024] NSWCCA 252 (DOCX, 23KB)
- `Lawlis v R [2025] NSWCCA 183 (3 November 2025).pdf` — (PDF, 12 pages, Chrome print-to-PDF from AustLII)

Both should be moved to `tests/fixtures/` during implementation.
