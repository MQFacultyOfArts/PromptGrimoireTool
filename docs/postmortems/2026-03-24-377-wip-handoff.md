### H2 Sub-Hypotheses (refined)

**Context:** Both timeout call sites (`_add_highlight` :288, `toggle_detail` :563) use `await ui.run_javascript(...)` with NiceGUI's default 1.0s timeout. Both await JS whose return values are never used (`removeAllRanges` returns void; `requestAnimationFrame` returns an unused int ID). The timeout fires when the browser main thread is busy and can't process the incoming WebSocket request within 1.0s.

**NiceGUI batching nuance:** NiceGUI accumulates all element changes during a server-side handler and sends them as one WebSocket message. Vue processes the batch in one render cycle. The MutationObserver (annotation-card-sync.js:103) likely fires **once** per batch, not per-card. The original "O(N²) from MutationObserver" claim needs verification — if Vue batches correctly, `positionCards` runs once per rebuild, doing O(N × T) work (N cards × T text nodes via linear scan), not O(N²). Needs instrumentation to confirm.

**H2a: The `await` calls are unnecessary and cause all the timeouts**
- `highlights.py:288`: `await ui.run_javascript("removeAllRanges")` — void return, no ordering dependency on subsequent server-side state changes (lines 290-294 are server-side only)
- `cards.py:563`: `await ui.run_javascript("rAF(_positionCards)")` — unused return, function ends after this call
- **Secondary bug:** when `removeAllRanges` times out, `TimeoutError` skips lines 290-294, leaving `selection_start`/`selection_end` stale and `highlight_menu` visible (ghost menu). The `finally` block resets `processing_highlight`, so the user can retry but with broken UI state.
- **Prediction:** removing `await` (fire-and-forget) eliminates ~5,500 of 5,947 JS timeouts AND fixes the stale-state secondary bug
- **Falsification:** if timeouts persist at other call sites at the same rate, browser main thread saturation is the deeper problem
- Evidence grade: **plausible** — mechanism is clear, both borders testable

**H2b: `positionCards()` linear scan is a secondary performance cost**
- `positionCards()` calls `charOffsetToRect(nodes, sc)` for each card, which calls `charOffsetToRange()` — a linear scan of the 5,020-element `textNodes` array. With N cards: O(N × 5020).
- **Prediction:** `positionCards()` with 190 cards takes >100ms
- **Falsification:** `console.time("positionCards")` instrumentation; if <50ms, not a meaningful contributor
- Evidence grade: **speculative** — untested
- **Fix path:** binary search in `charOffsetToRange` (textNodes are sorted by startChar)

**H2c: `applyHighlights()` full re-registration is a secondary performance cost**
- On every single highlight add/delete, `applyHighlights()` clears ALL CSS highlights then re-registers ALL from scratch. With 190 highlights, each calling `charOffsetToRange()` (linear scan of 5,020 nodes): O(190 × 5020) = ~950K iterations.
- **Prediction:** `applyHighlights()` takes >200ms on the Pabai workspace
- **Falsification:** `console.time("applyHighlights")` instrumentation
- Evidence grade: **speculative** — untested
- **Fix path:** incremental highlight updates (add/remove single highlight without full re-registration)

**Test approach for H2a:** `toggle_detail` is a closure inside `_build_annotation_card` — cannot be unit-tested by importing it. Tests must either go through `_build_annotation_card` (heavy) or be integration/E2E. `_add_highlight` is directly importable. Tests need to verify: (1) no TimeoutError propagation when JS is slow, (2) selection state cleaned up regardless.

## What's Next

### Diff-Based Cards (Design Needed)
`container.clear()` + full rebuild is O(n) at ~2.3–3.3ms per card. With 190 highlights = 431–632ms. Diff-based cards would make tag apply O(1). This is the primary performance improvement path — now demonstrated, not just hypothesised.

### Production Measurement
Deploy instrumentation and measure under real load. The local/production gap (566ms vs 3.3s page load) suggests significant non-server-side costs (WebSocket 1.8MB transit, network, pool contention). Individual `resolve_step` timings need re-capture (were suppressed by `setLevel(INFO)` bug).

### `setLevel(INFO)` Codebase Decision
~36 modules still have `logging.getLogger(__name__).setLevel(logging.INFO)`. Decision needed: remove all (makes DEBUG noisy globally) or keep and selectively remove when investigating specific modules.

### Deferred: MutationObserver batching verification
