# Ban User CLI Design

**GitHub Issue:** #102

## Summary

This document describes the implementation of a user ban system for PromptGrimoire. Instructors will be able to immediately suspend a student's account using a CLI command (`uv run grimoire admin ban <email>`), which needs to do two distinct things: set the ban state persistently so the user cannot log in again, and disconnect any browser tabs the user has open right now. Unbanning reverses both effects. There is no admin UI — the feature is intentionally CLI-only.

The challenge is that the CLI runs as a separate process from the live NiceGUI server, so a direct function call is not possible. The design bridges this with an internal HTTP endpoint (`POST /api/admin/kick`) on the server, secured by a shared secret. The CLI sets the ban in the database and in Stytch (the authentication provider), revokes all of the user's active login sessions, then calls that endpoint. The server receives the request, looks up the user's connected browser clients from an in-memory registry, navigates each to a `/banned` page, and disconnects the socket. A banned user who is not currently connected will be caught at their next login attempt by a check added to the session validation flow.

## Definition of Done

CLI commands (`uv run grimoire admin ban/unban <email>`) that immediately ban or unban a student. Banning sets `is_banned` on the local User model and in Stytch `trusted_metadata`, revokes all active Stytch sessions, and immediately disconnects all active NiceGUI clients belonging to that user via a client↔user registry. Banned users see an explicit "your account has been suspended" message on any page load or login attempt. Unbanning reverses the flag and allows normal login. No admin UI — CLI only.

## Acceptance Criteria

### ban-user-102.AC1: Ban command sets persistent state
- **ban-user-102.AC1.1 Success:** `admin ban <email>` sets `is_banned=True` and `banned_at` to current UTC time on User record
- **ban-user-102.AC1.2 Success:** `admin unban <email>` sets `is_banned=False` and `banned_at=None`
- **ban-user-102.AC1.3 Success:** `admin ban <email>` updates Stytch `trusted_metadata.banned` to `"true"`
- **ban-user-102.AC1.4 Success:** `admin unban <email>` clears Stytch `trusted_metadata.banned` to `""`
- **ban-user-102.AC1.5 Failure:** `admin ban <nonexistent@email>` exits with error, no DB or Stytch changes

### ban-user-102.AC2: Real-time client disconnection
- **ban-user-102.AC2.1 Success:** Client registry correctly tracks `user_id → client_id` mapping when user loads a page
- **ban-user-102.AC2.2 Success:** Client registry removes mapping when client is deleted (permanent disconnect)
- **ban-user-102.AC2.3 Success:** `disconnect_user()` navigates all of a user's active clients to `/banned` and disconnects them
- **ban-user-102.AC2.4 Success:** CLI ban command triggers kick endpoint, which calls `disconnect_user()` for the banned user

### ban-user-102.AC3: Banned user rejected on re-auth
- **ban-user-102.AC3.1 Success:** Banned user attempting session validation is redirected to `/banned` page
- **ban-user-102.AC3.2 Success:** `/banned` page displays "Your account has been suspended. Contact your instructor." with no navigation

### ban-user-102.AC4: Stytch session revocation
- **ban-user-102.AC4.1 Success:** `admin ban <email>` revokes all active Stytch sessions for the member
- **ban-user-102.AC4.2 Failure:** Ban command with user missing `stytch_member_id` warns but continues (local DB ban still applied)
- **ban-user-102.AC4.3 Success:** After ban, user cannot obtain a new Stytch session (Stytch metadata + revocation prevents re-auth)

### ban-user-102.AC5: List banned users
- **ban-user-102.AC5.1 Success:** `admin ban --list` displays all banned users with email, display name, and `banned_at` timestamp
- **ban-user-102.AC5.2 Success:** `admin ban --list` with no banned users shows empty result message

- **ban-user-102.AC6.2 Failure:** Request with incorrect bearer token returns 403

## Glossary

- **Stytch**: Third-party authentication provider used by PromptGrimoire. Handles magic link login, passkeys, and session management.
- **`trusted_metadata`**: Key-value store on a Stytch member record, controlled exclusively by the server. Used here to persist the ban flag on Stytch's side.
- **client registry**: In-memory dictionary (`user_id → {client_id, ...}`) tracking which open browser tabs belong to which authenticated user. Lost on server restart (acceptable — clients also disconnect on restart).
- **`page_route` wrapper**: Decorator on NiceGUI page handlers that performs authentication, binds logging context, and — after this change — registers the client in the registry.
- **`client.on_delete`**: NiceGUI lifecycle callback that fires when a client's connection is permanently closed (not temporary socket drops). Used for deregistration.
- **`hmac.compare_digest`**: Constant-time string comparison from Python stdlib, used to validate the shared secret on the kick endpoint to resist timing attacks.
- **Starlette route**: ASGI-level HTTP route mechanism used to add raw endpoints (like `/healthz` and `/api/admin/kick`) to the NiceGUI application outside the normal page system.
- **Rich**: Python library for terminal output. Used in the CLI to render tables and styled console messages.

## Architecture

Two-layer ban enforcement: CLI sets the ban state (local DB + Stytch), then signals the running NiceGUI server to disconnect the user immediately.

**Data layer:**
- `User.is_banned` (bool, default FALSE) and `User.banned_at` (datetime, nullable) columns on the existing User model
- `set_banned(user_id, is_banned)` DB function following the `set_admin()` pattern in `src/promptgrimoire/db/users.py`
- Stytch `trusted_metadata.banned` synced via the existing `_update_stytch_metadata()` helper in `src/promptgrimoire/cli/admin.py`
- Stytch session revocation via `client.sessions.revoke(member_id=stytch_member_id)` — revokes all sessions for the member at once

**Client registry:**
- Module-level `dict[UUID, set[Client]]` in `src/promptgrimoire/auth/client_registry.py` mapping `user_id → {Client, ...}` (stores NiceGUI Client objects, not string IDs, because `run_javascript()` requires the object reference)
- Registration: `page_route` wrapper in `src/promptgrimoire/pages/registry.py` registers `(user_id, client)` after reading `auth_user`
- Deregistration: `client.on_delete` callback removes the client (not `on_disconnect`, which fires on temporary socket drops)
- `disconnect_user(user_id)`: iterates the user's Client objects, calls `client.run_javascript('window.location.href = "/banned"', timeout=2.0)` on each. NiceGUI abstracts Socket.IO entirely — `app.sio` is not available. Cross-client navigation requires `run_javascript()`, not `ui.navigate.to()` (which only works for the current client context). Follows the proven `revoke_and_redirect()` pattern in `pages/annotation/broadcast.py`. Tolerates stale entries.

**CLI↔server bridge:**
- Internal `POST /api/admin/kick` endpoint on the NiceGUI server
- Secured by shared secret: `ADMIN_API_SECRET` env var, sent as `Authorization: Bearer <secret>`, validated via `hmac.compare_digest`
- On valid request, endpoint reads `is_banned` from DB (source of truth, doesn't trust request body), then calls `disconnect_user()` if banned
- CLI calls this endpoint via `httpx` after setting DB/Stytch state

**Ban enforcement on return:**
- `page_route` wrapper in `src/promptgrimoire/pages/registry.py` checks `is_banned` in DB for authenticated users, redirects to `/banned` if banned (centralised — all `page_route`-protected pages inherit the check automatically)
- `/banned` page: no-auth, `category="hidden"`, displays "Your account has been suspended. Contact your instructor."
- No per-page-load DB check — Stytch session revocation prevents re-auth, and real-time kick handles connected users

**Unban flow:**
- `set_banned(user_id, False)` clears `is_banned` and `banned_at`
- Stytch metadata cleared (`banned: ""`)
- User can log in normally — no kick endpoint call needed

## Existing Patterns

**`set_admin()` in `src/promptgrimoire/db/users.py`:** Direct precedent for `set_banned()`. Updates a boolean field on User by ID.

**`_cmd_instructor()` in `src/promptgrimoire/cli/admin.py`:** Direct precedent for the CLI commands. Pattern: `_require_user(email)` → DB update → `_update_stytch_metadata()` → Rich console output. The `ban`/`unban` commands follow this exactly, adding Stytch session revocation and the kick endpoint call.

**`/healthz` endpoint in `src/promptgrimoire/__init__.py`:** Precedent for adding Starlette routes. Uses `app.routes.insert(0, Route(...))`. The kick endpoint follows the same pattern.

**`search_worker.py` / `deadline_worker.py`:** Polling workers considered for ban detection but rejected — the internal HTTP endpoint gives immediate effect without polling latency.

**CRDT presence tracking:** The codebase already tracks per-client state for collaboration. The client↔user registry is a simpler version of the same concept — in-memory, lost on restart, which is fine because clients reconnect.

## Additional Considerations

**Stytch session revocation scope:** `client.sessions.revoke(member_id=...)` revokes all sessions across all devices. This is intentional — a ban should be total.

**Server restart:** The in-memory client registry is lost on restart. This is acceptable — all connected clients also disconnect on restart. When they reconnect and attempt to re-authenticate, the session validation ban check catches them (their Stytch sessions are already revoked).

**Admin self-ban prevention:** The CLI should warn (but not block) if the target email belongs to an admin. Admins can still be banned — the CLI just confirms intent.

**Source of truth:** The local database `is_banned` field is authoritative. Stytch metadata is best-effort sync. If a Stytch API call fails mid-ban, the CLI should warn but the local ban still takes effect. Session validation checks `is_banned` in DB, not Stytch. This means a partial Stytch failure degrades gracefully — the user is banned locally and caught on next login, even if their Stytch sessions weren't fully revoked.

**Missing `ADMIN_API_SECRET`:** If the secret is not configured, the kick endpoint must fail closed — return 503 and log an error. The CLI should also warn if the kick call fails (but the ban itself still succeeds in DB/Stytch). A ban without real-time kick is better than no ban at all.

**Race condition on kick:** A client could connect between Stytch session revocation and the kick endpoint call. This window is milliseconds. The session validation check on next page load catches them. Accepted risk.

**Deployment ordering:** Phases are sequential and must be deployed together. Partial-phase deployment (e.g., Phase 1 without Phase 2) would allow banning users who don't see the `/banned` page. Do not deploy individual phases to production.
