# Bulk Student Enrolment from XLSX

**GitHub Issue:** [#320](https://github.com/MQFacultyOfArts/PromptGrimoireTool/issues/320)

## Summary

Bulk Student Enrolment adds the ability to import student rosters from Moodle "Grades" XLSX exports directly into PromptGrimoire, replacing the current manual per-student enrolment process. A single file maps to three outcomes: User records are created or updated, CourseEnrollments are established, and tutorial group memberships are stored for future use. The implementation has two entry points — an admin CLI command for server-side batch operations and an upload widget on the Course Settings page for instructor self-service — but both share identical parsing and database logic with no duplication.

The design follows the functional core / imperative shell pattern already established by the wargame roster importer. A pure parser (`enrol/xlsx_parser.py`) converts raw bytes to validated `EnrolmentEntry` dataclasses with no database dependency, and a DB orchestration layer (`db/enrolment.py`) wraps all writes in a single atomic transaction. The whole pipeline is idempotent: re-importing the same XLSX is a no-op, student ID conflicts are treated as errors (requiring `--force` to override), and no Stytch API calls are made at import time — user accounts are provisioned lazily on first SSO login.

## Definition of Done

1. **CLI command** (`grimoire admin enroll-bulk`) imports a Moodle "Grades" XLSX file and bulk-enrols students into a course — creating Users (with `student_id`), CourseEnrollments, and printing a summary report. Idempotent on re-run.
2. **UI upload widget** on the Course Settings page for instructor self-service (same logic, different entry point).
3. **Schema changes:** `student_id` field on User, `student_group` and `student_group_membership` tables (Alembic migration). Groups column parsed and stored but not surfaced in UI.

**Success criteria:**
- Import the two real XLSX files (laws1000, laws8001) without errors
- Blank rows, trailing null columns, and `mq`-prefixed student IDs handled gracefully
- Re-running the same import is a no-op (upsert users, skip duplicate enrolments)
- Display names constructed from First name + Last name columns
- Groups column (`[foo], [bar]` format) parsed and stored in student_group/student_group_membership
- No Stytch API calls — JIT provisioning handles member creation on SSO login

**Out of scope:**
- Stytch member pre-creation (JIT provisioning instead)
- UI for viewing/managing groups (data stored, not surfaced)
- Group-based filtering or wargame team auto-population from groups

## Glossary

- **Moodle**: Open-source LMS used by Macquarie University. The XLSX files being parsed are exported from Moodle's Gradebook as "Grades" exports.
- **CourseEnrollment**: SQLModel join record linking a User to a Course with an assigned role (e.g. `student`). Distinct from Stytch RBAC.
- **StudentGroup / StudentGroupMembership**: Two new schema tables. `StudentGroup` names a tutorial group within a course; `StudentGroupMembership` is the many-to-many join linking users to groups.
- **student_id**: New nullable field on the User model holding the university student identifier (e.g. `mq12345678`). Separate from Stytch's `external_id`.
- **Functional core / imperative shell**: Architecture pattern where pure functions handle logic/validation and a separate shell layer handles I/O (database writes, file reads).
- **openpyxl**: Python library for reading and writing Excel XLSX files.
- **JIT provisioning**: Just-in-time user creation — Stytch member records created on first SSO login, not pre-populated from the roster.
- **Upsert**: Database write that inserts a new row or updates an existing one based on unique constraint match. Makes re-imports idempotent.
- **EnrolmentEntry**: Frozen dataclass representing one parsed XLSX row — the contract between functional core and imperative shell.
- **EnrolmentReport**: Dataclass returned by `bulk_enrol()` summarising outcome: counts of created/skipped users, enrolments, groups, memberships, and student ID conflicts.
- **EnrolmentParseError**: Exception raised by `parse_xlsx()` on validation failure. Carries line-numbered error strings.
- **StudentIdConflictError**: Exception raised by `bulk_enrol()` when a user's existing `student_id` differs from the XLSX value and `force=False`. Lists all conflicts for display.
- **Rich**: Python terminal formatting library for structured CLI output.
- **Alembic**: Schema migration tool for the project's PostgreSQL database.
- **`ui.upload()`**: NiceGUI file upload component. Handler receives `UploadEventArguments` with raw bytes.
- **`[foo], [bar]` format**: Bracket-delimited group name format in Moodle's Groups column. Parser strips brackets and splits on `, `.

## Architecture

Functional core / imperative shell, mirroring the wargame roster ingestion pattern (`wargame/roster.py` + `db/wargames.py`).

**Functional core** — `src/promptgrimoire/enrol/xlsx_parser.py`:
- Pure function: bytes in, structured data out (or structured errors)
- Parses Moodle "Grades" XLSX exports via openpyxl
- Validates emails, constructs display names from first/last name columns
- Parses Groups column (`[foo], [bar]` bracket-delimited format)
- Returns `list[EnrolmentEntry]` or raises `EnrolmentParseError` with line-numbered errors
- No database, no IO beyond the bytes it receives

**Contracts:**

```python
@dataclass(frozen=True)
class EnrolmentEntry:
    email: str
    display_name: str
    student_id: str
    groups: list[str]  # parsed from "[foo], [bar]" format; empty if column absent

class EnrolmentParseError(Exception):
    errors: list[str]  # human-readable, line-numbered

def parse_xlsx(data: bytes) -> list[EnrolmentEntry]: ...
```

**Imperative shell** — `bulk_enrol()` in `src/promptgrimoire/db/enrolment.py`:
- Own module (parallels `db/wargames.py` having its own file rather than living in general CRUD)
- Single atomic transaction wrapping all writes
- For each entry: `find_or_create_user()` (upsert), set `student_id` (conflict-aware), `enroll_user()` (skip duplicates)
- Create `StudentGroup` records per course, `StudentGroupMembership` linking users to groups
- Student ID conflicts are **errors by default** — transaction aborts with `StudentIdConflictError`
- `force=True` parameter overrides: overwrites conflicting student_id and logs warning
- Returns `EnrolmentReport` with counters

```python
class StudentIdConflictError(Exception):
    """Raised when a user's existing student_id differs from the import."""
    conflicts: list[tuple[str, str, str]]  # (email, existing_id, new_id)

@dataclass
class EnrolmentReport:
    entries_processed: int
    users_created: int
    users_existing: int
    enrolments_created: int
    enrolments_skipped: int
    groups_created: int
    group_memberships_created: int
    student_ids_overwritten: int  # only non-zero when force=True

def bulk_enrol(
    entries: list[EnrolmentEntry],
    course_id: UUID,
    role: str = "student",
    force: bool = False,
) -> EnrolmentReport: ...
```

**Entry points:**
- CLI: `grimoire admin enroll-bulk` in `src/promptgrimoire/cli/admin.py`
- UI: `ui.upload()` widget in `src/promptgrimoire/pages/courses.py` settings dialog

Both call `parse_xlsx()` then `bulk_enrol()` — no logic duplication.

## Existing Patterns

**Wargame roster ingestion** (`wargame/roster.py` + `db/wargames.py`):
- CSV parsing as pure function returning frozen dataclasses
- Structured errors with line numbers
- Case-insensitive header matching
- Email validation (one `@`, non-empty local/domain)
- Duplicate email detection across rows
- Atomic DB transaction with upsert semantics
- `RosterReport` dataclass for summary output

This design follows all of these patterns. Divergences:
- XLSX via openpyxl instead of CSV via stdlib `csv`
- Display name constructed from two columns (roster.py doesn't handle names)
- Groups column parsing (no equivalent in roster.py)
- Target is CourseEnrollment (not wargame team ACL)

**File upload** (`pages/annotation/content_form.py` + `upload_handler.py`):
- `ui.upload()` with `auto_upload=True`, file type restriction via `.props('accept=...')`
- Handler receives `UploadEventArguments`, reads bytes via `await upload_event.file.read()`
- Results displayed via `ui.notify()` with type flags

**CLI pattern** (`cli/admin.py`):
- Async handler prefixed `_cmd_*`
- Sync wrapper with `@admin_app.command()` calling `asyncio.run()`
- Rich table output for structured results
- Exit code 1 on errors

## Additional Considerations

**Student ID normalisation:** The `mq` prefix on some IDs is preserved as-is. Normalisation (e.g. stripping prefix for Stytch `external_id` mapping) is a consumer concern, not a parser concern. Different downstream systems may want different formats.

**Groups column absence:** If the XLSX has no Groups column, `EnrolmentEntry.groups` is an empty list. The parser does not fail — groups are optional. Group-related DB operations are skipped when no entries have groups.

**Future: Stytch external_id sync:** When needed, a separate command can iterate Users with `student_id` set and call the Stytch API to set `external_id`. This is independent of the bulk enrolment flow.
