# PDF Export Queue Design

**GitHub Issue:** #402

## Summary

PDF export currently runs synchronously inside the NiceGUI page handler: the server spawns a pandoc/LaTeX pipeline that can take up to 85 seconds, and the result is lost if the user's browser connection drops or times out during that window. This design replaces that approach with a database-backed job queue. When a user requests an export, the page handler immediately inserts an `ExportJob` record and returns — no compilation happens in the request context. A background worker picks up queued jobs, runs the existing pandoc → LaTeX → PDF pipeline, and records the result (or error) back to the database. The user's browser polls for status every two seconds and displays a progress indicator; when the job completes, a download button appears.

Downloads are served through a raw FastAPI route using a time-limited token rather than through NiceGUI's session machinery. This means the download works even if the user has disconnected and reconnected between submitting the export and clicking the button. Tokens are multi-use within their 24-hour TTL, allowing repeat downloads without re-export. Concurrent exports from the same user are rejected at the database level before any process is spawned, preventing the retry-spam pattern that previously degraded the shared connection pool. NiceGUI's `response_timeout` is raised from 3 to 60 seconds as a permanent fix for #377 Finding 6 — slow page loads should not delete clients.

## Definition of Done

1. **PDF exports complete and deliver regardless of client lifecycle.** An 85s compile results in a downloadable PDF even if the NiceGUI client disconnects/reconnects during compilation.

2. **Progress feedback in the UI** shows export stages (queued → pandoc → compiling → ready). When ready, a "Download your PDF" button appears — visible only to the authenticated user who initiated the export.

3. **Download served via raw FastAPI route** with `secrets.token_urlsafe` token — no NiceGUI session dependency. User-facing UX is just the download button.

4. **Compile queue with per-user concurrency limit** — FIFO ordering, semaphore(2) concurrency cap. Concurrent exports from the same user are rejected (one active job per user).

5. **Increase `response_timeout` to 60s** on all `page_route` pages — permanent fix for #377 Finding 6, preventing client deletion on slow page loads.

6. **Retry-spam no longer cascades** connection pool degradation across all users.

- **export-queue-402.AC2.2 Success:** Timer deactivates after job reaches terminal state (completed or failed)
- **export-queue-402.AC2.3 Edge:** Page reload during active export re-renders progress indicator for running job or download button for completed job

### export-queue-402.AC3: Download via FastAPI route
- **export-queue-402.AC3.1 Success:** Clicking download button triggers file download via `/export/{token}/download`
- **export-queue-402.AC3.2 Success:** Token is multi-use — repeat downloads within 24-hour TTL return the same PDF
- **export-queue-402.AC3.3 Failure:** Request with expired token (>24 hours) returns 404
- **export-queue-402.AC3.4 Failure:** Request with nonexistent token returns 404

### export-queue-402.AC4: Queue with per-user concurrency limit
- **export-queue-402.AC4.1 Success:** Two users submit exports — both are processed (concurrency cap 2)
- **export-queue-402.AC4.2 Success:** Three users submit exports — first two run concurrently, third waits, gets processed when a slot frees
- **export-queue-402.AC4.3 Failure:** User with an active export submits a second — rejected with "already in progress" notification

### export-queue-402.AC5: response_timeout fix (#377)
- **export-queue-402.AC5.1 Success:** `page_route` decorator passes `response_timeout=60` to `ui.page()`

### export-queue-402.AC6: No cascade from retry-spam
- **export-queue-402.AC6.1 Success:** User's concurrent export attempt is rejected at the database level — no LaTeX process spawned, no pool impact
- **export-queue-402.AC6.2 Success:** Expired jobs and PDF files are cleaned up within 24 hours

## Glossary

- **NiceGUI**: Python web UI framework wrapping FastAPI. Manages browser-server state via WebSocket. Each connected browser tab is a "client"; NiceGUI can delete client state if a response takes too long (controlled by `response_timeout`).
- **client lifecycle**: NiceGUI's state associated with a single browser connection. A client can disconnect and reconnect. If NiceGUI decides a client is gone, it deletes its state — which previously caused in-flight exports to become unreachable.
- **ExportJob**: The new SQLModel database table introduced by this design. Stores one row per export request, tracking its status, payload, and result.
- **FOR UPDATE SKIP LOCKED**: PostgreSQL locking clause used when claiming a job from the queue. Lets multiple worker processes each grab a different row atomically without blocking each other.
- **Per-user concurrency limit**: Each user can have at most one active (queued or running) export job. Combined with FIFO ordering, this makes the queue inherently fair across users.
- **Semaphore (compile semaphore)**: `asyncio.Semaphore(2)` capping concurrent LaTeX compilations. Pre-existing in `export/pdf.py`; the worker acquires it before calling `compile_latex()`.
- **pandoc**: Document conversion tool converting annotated workspace content to LaTeX before PDF compilation.
- **latexmk / lualatex**: LaTeX toolchain compiling `.tex` files to PDF. The slow step (up to 85 seconds for complex documents).
- **FastAPI**: HTTP framework NiceGUI is built on. Raw FastAPI routes handle requests outside NiceGUI's session machinery — used here for the download endpoint.
- **`secrets.token_urlsafe`**: Python stdlib function generating a cryptographically random URL-safe string. Produces the download token (256 bits of entropy).
- **`ui.timer`**: NiceGUI component calling a callback on a repeating interval. Polls job status every 2 seconds.
- **`ui.download()`**: NiceGUI helper triggering a browser file download. Initiates download via the token URL when the user clicks the button.
- **`response_timeout`**: NiceGUI parameter on `ui.page()` controlling how long a handler can run before the client is deleted. Currently 3s; raised to 60s permanently (#377 Finding 6).
- **`page_route` decorator**: Project-internal decorator in `pages/registry.py` wrapping `ui.page()` with ban checking and middleware.
- **polling worker**: Pattern in `deadline_worker.py` and `search_worker.py` — async loop via `asyncio.create_task()` querying the database periodically. The export worker follows this pattern.
- **TTL (time-to-live)**: 24-hour expiry on download tokens and PDF files. After expiry, tokens return 404 and files are deleted.
- **connection pool degradation**: The production failure mode being fixed. Retry-spam exhausts the database connection pool and makes the application unresponsive for all users.

## Architecture

PDF export is decoupled from the NiceGUI page handler by introducing a database-backed job queue. The page handler gathers data from the live client context and inserts an `ExportJob` row. A background worker polls for queued jobs, runs the pandoc/LaTeX pipeline, and writes results back to the database. The user's browser polls job status via `ui.timer` and presents a download button on completion. Download is served by a raw FastAPI route using a time-limited token — no NiceGUI client dependency.

### ExportJob Table

New SQLModel table (via Alembic migration):

| Column | Type | Purpose |
|--------|------|---------|
| `id` | UUID (PK) | Job identifier |
| `user_id` | UUID (FK → User) | Who requested the export |
| `workspace_id` | UUID (FK → Workspace) | Source workspace |
| `status` | Enum: `queued`, `running`, `completed`, `failed` | Job lifecycle state |
| `payload` | JSON | Self-contained export inputs (highlights, html_content, notes_latex, tag_colours, filename, word count params) |
| `download_token` | str (nullable) | `secrets.token_urlsafe(48)`, set on completion |
| `token_expires_at` | datetime (nullable) | Token expiry (24 hours after completion) |
| `pdf_path` | str (nullable) | Filesystem path to compiled PDF |
| `error_message` | str (nullable) | Error detail if failed |
| `created_at` | datetime | Insertion timestamp, used for FIFO ordering |
| `started_at` | datetime (nullable) | When worker claimed the job |
| `completed_at` | datetime (nullable) | When compile finished or failed |

### Data Flow

```
User clicks "Export PDF"
  → Page handler: validate, extract data, check word count (AC5/AC6)
  → INSERT ExportJob(status='queued', payload={...})
  → Start ui.timer(2s) polling job status

Export Worker (background task):
  → SELECT ... FOR UPDATE SKIP LOCKED (fair scheduling)
  → UPDATE status='running'
  → pandoc → tex → latexmk (existing pipeline)
  → UPDATE status='completed', set download_token, pdf_path

ui.timer callback:
  → Query ExportJob.status
  → On 'completed': dismiss spinner, show "Download your PDF" button
  → On 'failed': dismiss spinner, show error notification
  → Deactivate timer on terminal state

User clicks download button:
  → ui.download("/export/{token}/download")
  → FastAPI route validates token, serves FileResponse
```

### Job Claiming (FIFO with Per-User Concurrency Limit)

Per-user rejection ensures at most one job per user in the queue, so FIFO ordering is inherently fair across users. The worker claims jobs using a query that deprioritises users with running jobs as belt-and-suspenders:

```sql
SELECT * FROM export_job
WHERE status = 'queued'
ORDER BY (
    SELECT COUNT(*) FROM export_job ej2
    WHERE ej2.user_id = export_job.user_id
    AND ej2.status = 'running'
) ASC, created_at ASC
FOR UPDATE SKIP LOCKED
LIMIT 1
```

### Per-User Rejection

Before inserting a new job, the page handler checks for existing `queued` or `running` jobs for the same user. If found, the export is rejected with a notification: "A PDF export is already in progress."

### Download Route

Raw FastAPI route registered on the NiceGUI app:

```python
@app.get("/export/{token}/download")
async def download_export(token: str) -> FileResponse: ...
```

Validates token exists and has not expired. Serves `FileResponse(pdf_path)`. Multi-use within 24-hour TTL — allows repeat downloads without re-export. Token is `secrets.token_urlsafe(48)` (256 bits, unguessable). The user never sees the URL directly — `ui.download()` triggers it behind the "Download your PDF" button.

### response_timeout Fix (#377 Finding 6)

`page_route` in `registry.py` currently uses NiceGUI's default 3.0s `response_timeout`. This is increased to 60s permanently — slow page loads (DB queries, CRDT loading) should not delete clients. This is a fix for #377 Finding 6, not a temporary stopgap.

## Existing Patterns

### Background Workers

`deadline_worker.py` and `search_worker.py` both use the same pattern: an async polling loop launched via `asyncio.create_task()` in `__init__.py`, using `get_session()` for database access. The export worker follows this pattern exactly.

### Per-User Export Lock

`pages/annotation/pdf_export.py` already implements a per-user `asyncio.Lock` and rejection of concurrent exports. This logic moves to the database layer (check for existing `queued`/`running` jobs) so it works across server restarts and future multi-server deployments.

### Compile Semaphore

`export/pdf.py` has `_compile_semaphore = asyncio.Semaphore(2)` capping concurrent LaTeX compilations. This remains unchanged — the worker acquires the semaphore before calling the existing `compile_latex()` function.

### FastAPI Routes on NiceGUI App

`cli/e2e/_server_script.py` registers raw FastAPI routes via `@app.get(...)` on the NiceGUI `app` object. The download route follows this established pattern.

### No New Patterns Introduced

All components follow existing codebase patterns. The only new element is the `ExportJob` table, which follows the established SQLModel + Alembic convention.

## Additional Considerations

**Future server split:** The export worker can be extracted to a separate server by adding a CLI entry point (`uv run grimoire worker export`) that runs the same worker loop standalone. The ExportJob table and FOR UPDATE SKIP LOCKED query work identically across processes. Tracked in #404.

**JSON payload size:** The payload column stores highlights, HTML content, notes LaTeX, and tag colours. For a typical legal annotation workspace this is 50-200 KB — well within PostgreSQL's JSON column capacity. Cleanup after 24 hours prevents accumulation.

**Interaction with existing per-user lock:** The in-memory `asyncio.Lock` in `pdf_export.py` is replaced by the database-level per-user check. The database check works across restarts and future multi-server deployments. The in-memory lock is removed.
