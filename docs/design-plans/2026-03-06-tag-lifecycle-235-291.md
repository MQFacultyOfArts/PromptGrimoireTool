# Tag Lifecycle Refactor Design

**GitHub Issue:** #235, #291 (consolidates #237, #240, #241, #242, #243, #280, #283)

## Summary

PromptGrimoire currently stores tag and tag group metadata exclusively in PostgreSQL, which means tag changes — create, rename, recolour, reorder — require a manual page refresh to appear on other connected clients. Several related bugs compound this: colour changes silently fail to save, the organise tab does not propagate drag operations, and tag import is restricted to instructors. This refactor addresses all of these by making the pycrdt collaborative document the authoritative source for live rendering.

The approach is dual storage: PostgreSQL remains canonical for export, full-text search, and queries, while a pair of new Maps in the CRDT document (`tags` and `tag_groups`) carry the metadata needed for live UI. Every mutation writes to the database first, then updates the CRDT. The existing CRDT observer/broadcast mechanism then automatically propagates changes to all connected sessions with no additional infrastructure. On workspace load, a consistency check hydrates the CRDT from the database if it is empty, and reconciles any drift if both have data. The existing `tag_order` Map is folded into the new structure, the fragile save-on-blur event pattern in the management dialog is replaced with NiceGUI data bindings, and tag import is opened to all users against any workspace they can read.

## Definition of Done

1. **Tag metadata lives in both DB and CRDT.** The CRDT document contains a `tags` Map and a `tag_groups` Map with all metadata needed for rendering (name, colour, group reference, order, highlight list). The DB remains canonical for export, FTS, and queries. Python server writes to both on every mutation.

2. **Tag lifecycle is complete and sync'd.** Creating, editing, or deleting a tag/group via any UI path (quick create, management dialog, import) writes to DB + CRDT. The existing CRDT sync propagates changes to all connected clients. No tag ever appears "uncategorised" unless explicitly ungrouped. No refresh required.

3. **Unified import from any accessible workspace.** A single import mechanism replaces the current instructor-only `import_tags_from_activity()`. Any user can import tags/groups from any workspace they have at least read access to. Additive merge, preserve ordering, skip duplicates by name.

4. **Tag colour persistence works.** Colour changes via the management dialog save correctly (fix blur/change event handling) and propagate via CRDT sync.

5. **Organise tab fully sync'd.** Tag reordering, group assignment changes, and drag operations update the CRDT, propagating to all clients. The `tag_order` map is folded into the `tags` Map structure.

6. **Existing contracts preserved.** Export pipeline, FTS extraction, LaTeX preamble, and highlight CSS all continue to work — they read from DB or from `TagInfo` (which can be hydrated from either source as appropriate).

## Glossary

- **CRDT (Conflict-free Replicated Data Type)**: A data structure that can be edited concurrently by multiple clients and merged automatically without conflicts. PromptGrimoire uses the `pycrdt` library, which implements the Yjs CRDT protocol, to synchronise annotation state across browser sessions in real time.

- **pycrdt**: A Python implementation of the Yjs CRDT library. Provides `Map`, `Array`, and `Text` types that can be composed into a document (`Doc`), serialised to bytes for storage, and broadcast as update diffs to connected clients.

- **`AnnotationDoc`**: The application's wrapper around a pycrdt `Doc`. Holds all collaborative state for one workspace: highlights, tag order, client cursor metadata, and (after this refactor) tag and tag group metadata.

- **CRDT Map**: A pycrdt key-value store analogous to a Python `dict`. Changes to a Map are captured as CRDT update diffs and broadcast to all observers. In this design, `tags` and `tag_groups` are top-level Maps in `AnnotationDoc` keyed by UUID string.

- **CRDT observer**: A callback registered on a pycrdt `Doc` or Map that fires whenever a change is applied, including changes received from remote clients. Used here to trigger UI rebuilds on all connected sessions without explicit refresh.

- **`TagInfo`**: A frozen Python dataclass that packages tag metadata (name, colour, group, order) for downstream consumers in the annotation page — the tag toolbar, highlight CSS generation, and the organise tab. After this refactor, it is hydrated from CRDT maps rather than a DB query during live sessions.

- **`tag_order` Map**: The existing CRDT Map that associates each tag UUID with an ordered list of highlight IDs. This refactor folds it into the `highlights` field of each entry in the new `tags` Map, then removes it.

- **Dual write**: The pattern of writing to both the database and the CRDT document in a single mutation. The database write happens first (authoritative persistence); the CRDT write follows within a transaction batch to keep state consistent.

- **`doc.transaction()`**: A pycrdt context manager that batches multiple CRDT writes into a single update diff. This ensures all fields of a tag are delivered to remote clients atomically, rather than as a sequence of partial updates.

- **Hydration**: The startup-time process of populating the CRDT `tags`/`tag_groups` Maps from the database when the CRDT document has no tag data yet. Covers existing workspaces created before this refactor.

- **NiceGUI data binding (`bind_value()`)**: A NiceGUI mechanism that links a UI component's value to a Python variable. When the variable changes, NiceGUI pushes the update to the browser automatically — eliminating the need for manual `on("blur")` / `on("change")` event handlers.

- **Save-on-blur**: The event-driven pattern currently used in `tag_management_rows.py` where a form field saves its value when it loses focus (`blur` event) or when a change event fires. Fragile because some input widgets (notably colour pickers) do not reliably emit these events. Replaced by data bindings in this refactor.

- **FTS (Full-Text Search)**: PostgreSQL's built-in full-text search. PromptGrimoire materialises workspace content (highlights, tags, comments) into a `search_text` column and maintains GIN indexes against it. Tag metadata for FTS continues to come from the database, not the CRDT.

- **`resolve_permission()`**: An ACL utility in `auth/__init__.py` that determines what access level (owner, editor, viewer, etc.) a given user has on a workspace. The unified import mechanism uses this to gate source workspace access.

- **Additive merge**: An import strategy that only adds new entries and never modifies or removes existing ones. Tags from the source workspace are added to the target; any tag whose name already exists in the target is skipped.

- **SQLModel**: The ORM used in PromptGrimoire, combining Pydantic data validation with SQLAlchemy persistence. `tag` and `tag_group` tables are defined as SQLModel classes.

## Architecture

### Data Model: Dual Storage

Tag and tag group metadata lives in two places:

- **PostgreSQL** (canonical): `tag` and `tag_group` tables via SQLModel. Authoritative for export, FTS, and queries. Written first on every mutation.
- **pycrdt document** (rendering + sync): `tags` and `tag_groups` Maps within `AnnotationDoc`. Authoritative for live rendering. Written second on every mutation. Propagates to all connected clients via existing CRDT observer/broadcast.

### CRDT Schema

Two new top-level Maps in `AnnotationDoc`, replacing the existing `tag_order` Map:

**`tags` Map** — keyed by tag UUID string:
```python
{
    "uuid-string": {
        "name": str,           # display name
        "colour": str,         # hex e.g. "#1f77b4"
        "group_id": str | None, # references tag_groups key
        "description": str | None,
        "order_index": int,
        "highlights": list[str], # ordered highlight IDs (folded from tag_order)
    }
}
```

**`tag_groups` Map** — keyed by group UUID string:
```python
{
    "uuid-string": {
        "name": str,
        "colour": str | None,  # group header background hex
        "order_index": int,
    }
}
```

The existing `highlights` Map is unchanged — each highlight still has a `"tag"` field referencing a tag UUID.

### Mutation Flow

All tag/group mutations follow one path:

1. DB write (canonical persistence, existing `db/tags.py` functions)
2. CRDT write (update `tags`/`tag_groups` Map, using `doc.transaction()` to batch)
3. pycrdt observer fires automatically
4. All connected sessions receive update, rebuild `TagInfo` from CRDT maps
5. NiceGUI data bindings propagate changes to browser — targeted updates, not full rebuilds

### Rendering Source

**CRDT-primary.** `workspace_tags()` reads from CRDT maps instead of querying the DB. Returns the same `TagInfo` frozen dataclass. All downstream consumers (`tag_colours()`, `_build_tag_toolbar()`, `_build_highlight_pseudo_css()`) are unchanged.

Export pipeline and FTS continue reading from DB — they operate outside live CRDT sessions.

### Consistency Defence

On every client connect (not just workspace creation), two checks run:

1. **Hydration**: If CRDT `tags`/`tag_groups` Maps are empty but DB has rows, populate CRDT from DB. This guarantees `workspace_tags()` never returns empty tags for a workspace that has tags in the DB.
2. **Consistency**: If both have data, verify agreement. DB is authoritative — missing CRDT entries are added, orphaned CRDT entries are logged (WARNING level) and removed.

These are permanent defence-in-depth measures against network drops or partial write failures, not migration artefacts. They run on every client connect so that partial failures during a live session (DB write succeeds, CRDT write fails) are repaired when the next client joins, rather than requiring full workspace eviction and reload.

**Accepted failure mode**: If a dual write partially fails (DB succeeds, CRDT fails), the tag exists in the DB but is invisible to connected clients until either (a) another CRDT write triggers the observer, or (b) a new client connects and the consistency check repairs the CRDT. This window is acceptable for a classroom context where tag mutations are infrequent.

### Unified Import

A single import function replaces `import_tags_from_activity()`:

- **Source**: Any workspace the user has at least read access to (via `resolve_permission()`)
- **Operation**: Read source workspace's tags/groups from DB → create new UUIDs → write to target DB + CRDT
- **Merge rules**: Additive. Skip tags with duplicate names. Preserve ordering (offset by target's current max). No locked status imported (defaults to unlocked — that's the owner's choice)
- **UI**: Workspace picker in tag management dialog, available to all users. Source workspaces grouped by course/unit plus personal workspaces.

### UI Binding Strategy

Tag management dialog and toolbar use NiceGUI data bindings instead of manual save-on-blur event handlers. This eliminates the class of bugs where event handlers don't fire (colour picker `on("change")` issue in #242/#280). The observer callback updates bound values, which NiceGUI propagates to the browser without full component rebuilds.

## Existing Patterns

### Followed

- **CRDT document structure**: New `tags`/`tag_groups` Maps follow the same pattern as existing `highlights` and `client_meta` Maps — top-level Maps keyed by UUID string, values as dicts. Full-replacement pattern for updates (read dict, mutate, write back).
- **DB + CRDT dual write**: Existing `delete_tag()` already writes to both DB and CRDT (`_cleanup_crdt_highlights_for_tag()`). This design extends that pattern to all mutations.
- **`TagInfo` dataclass**: The frozen dataclass contract is preserved. Only the data source changes (CRDT instead of DB query).
- **Permission resolution**: Import uses existing `resolve_permission()` for access checks.

### Diverged

- **`tag_order` Map eliminated**: Currently a separate top-level Map (`{tag_uuid: Array([highlight_ids])}`). Folded into the `tags` Map as each tag's `highlights` field. Reduces top-level Maps and keeps tag data co-located.
- **`_refresh_tag_state()` becomes observer-driven**: Currently a manually-called function after each mutation. Replaced by CRDT observer callback that fires automatically on all sessions. Eliminates the pattern of "write to DB, then remember to call refresh."
- **Save-on-blur replaced by data bindings**: The `on("blur")` / `on("change")` event pattern in `tag_management_rows.py` is replaced by NiceGUI `bind_value()`. This is a deliberate divergence to eliminate the colour persistence bugs.

## Additional Considerations

**Consistency check cost:** The hydration/consistency check on workspace load compares CRDT maps against DB rows. For workspaces with many tags this is two DB queries plus an in-memory comparison — negligible. The check should log discrepancies at WARNING level for monitoring.

**CRDT document size:** Tag metadata adds ~200-300 bytes per tag to `crdt_state`. A workspace with 50 tags adds ~15KB. This is small relative to highlight data and should not impact sync performance.
