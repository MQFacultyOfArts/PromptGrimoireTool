## Human Verification Required

| AC | Criterion | Justification | Verification Approach |
|----|-----------|---------------|----------------------|
| epoch-analysis.AC1.1 | `github` command fetches merged PRs within time window and stores in `github_events` table | End-to-end integration with the live GitHub API cannot be tested in CI without credentials and network access. The unit tests mock httpx responses but cannot verify pagination behaviour, rate limiting, or real API response schema drift. | Run `incident_db.py github --start "2026-03-15 00:00" --end "2026-03-18 20:37" --db /tmp/test_github.db` against the real repo. Verify `sources` shows a "github" format row. Verify event count is non-zero and matches approximate PR merge count for the window. |
| epoch-analysis.AC1.2 | Token resolution falls back from `GITHUB_TOKEN` env to `gh auth token` subprocess | Subprocess invocation of `gh auth token` depends on the user's local gh CLI installation state, which varies per machine and cannot be reliably mocked in CI. | With `GITHUB_TOKEN` unset, run the `github` command and verify it succeeds using `gh auth token`. Then unset both and verify the clear error message (AC1.4 UAT). |
| epoch-analysis.AC2.1 | Commit hash transitions correctly identify epoch boundaries | Unit tests use synthetic data. Validating against the real 3.6M-event `incident.db` confirms the SQL works at scale and the epoch count matches the known deployment history. | Run `extract_epochs(conn)` against `incident.db`. Verify epoch count matches the known ~12 deployments in the review window. Spot-check 2-3 commit hashes against `git log`. |
| epoch-analysis.AC3.2 | HAProxy p50/p95/p99 response times | SQLite percentile approximation via OFFSET is an approximation, not exact. Unit tests verify the mechanism works but cannot confirm the approximation is acceptable on production-scale data. | Run `query_epoch_haproxy()` against a long epoch in `incident.db`. Compare p50/p95/p99 values against manual SQL verification (`ORDER BY ta_ms` spot check). Confirm `sample_count` is reported alongside percentiles. |
| epoch-analysis.AC5.1 | Trend deltas correctly identify known incidents | Unit tests verify arithmetic correctness with synthetic data but cannot confirm the tool surfaces the known Mar 15 crash loop and pool_size=80 config change from the real dataset. | Run full trend analysis against `incident.db`. Verify the Mar 15 crash-bounce epochs are flagged. Verify the pool_size=80 epoch shows a meaningful change in error rate or resource usage vs its predecessor. |
| epoch-analysis.AC6.1 | `review` command produces complete markdown report | The full orchestration against real data involves all phases interacting. Unit tests mock individual functions but cannot verify the assembled report is coherent, well-formatted, and renders correctly in a markdown viewer. | Run `incident_db.py review --db incident.db --counts-json counts.json --output /tmp/review.md`. Open in a markdown viewer. Check: all sections present, tables render correctly, epoch data is plausible, trend anomaly flags appear where expected, crash-bounces marked. |
