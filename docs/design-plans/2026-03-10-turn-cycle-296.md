# Wargame Turn Cycle Engine Design

**GitHub Issue:** #296

## Summary

The Wargame Turn Cycle Engine (Seam 3) is the server-side state machine that drives a tabletop-style simulation game in which student teams submit written orders each round, receive AI-generated responses from a game master perspective, and progress through successive rounds until the scenario ends. The engine is headless — it has no UI of its own and exposes a service layer that the Player Room (Seam 4) and GM Control (Seam 5) will call into.

The implementation follows the functional core / imperative shell architecture already established in Seam 2. Pure domain logic (deadline calculation, CRDT text extraction, prompt assembly) lives in `wargame/turn_cycle.py` and `wargame/agents.py` with no side effects. The imperative shell in `db/wargames.py` orchestrates database writes, AI calls, and scheduler interactions. A hard deadline is enforced by APScheduler: when the timer fires, all team move buffers are locked simultaneously and the AI preprocessing pipeline runs serially per team, producing a draft GM response for each. The GM then reviews and publishes the round, which generates student-facing summaries, advances the round counter, clears move buffers, and schedules the next deadline. The full cycle repeats until `end_simulation()` is called.

## Definition of Done

Seam 3 delivers the headless turn cycle engine — all server-side mechanics that Seam 4 (Player Room) and Seam 5 (GM Control) call into. Specifically:

1. **CRDT storage** — `move_buffer_crdt` and `notes_crdt` bytea columns on WargameTeam, plus Alembic migration.
2. **Timer management** — APScheduler 3.11.x integration with SQLAlchemy job store: schedule deadline on publish, fire lock when deadline arrives (including after app restart). `misfire_grace_time=None` ensures missed deadlines always fire.
3. **Hard-deadline lock** — Lock all teams' move buffers simultaneously when timer fires, transition `round_state` from `drafting` → `locked`.
4. **Snapshot pipeline** — Extract markdown from each team's CRDT move buffer; empty/whitespace → `"No move submitted"`.
5. **AI pre-processing** — For each team (serial): PydanticAI call (Sonnet 4.6) with cadet orders + game-state artifact + full turn history → draft assistant response + updated game-state artifact.
6. **Publish pipeline** — GM-triggered "publish all": release approved responses, generate student summaries (separate AI call per team), advance round counter, start next timer.
7. **Completion gating** — Next round cannot open until all teams have published responses.
8. **One-response invariant** — Exactly one assistant response per team per round, enforced in service logic.

**Success criteria:**
- All functions tested (unit + integration with DB).
- APScheduler misfire recovery verified (jobs scheduled in the past fire on startup).
- PydanticAI integration tested with mock/stub model responses.
- CRDT snapshot extraction tested with real pycrdt documents.
- Round state transitions tested exhaustively (drafting → locked → drafting cycle).
- One-response invariant cannot be violated under any code path.

**Out of scope:**
- Player room UI (Seam 4).
- GM control UI (Seam 5).
- Projection and tagging (Seam 6).
- End-state and export (Seam 7).
- Bounded concurrency for AI calls (MVP is serial).
- Context summarisation/compression.

## Glossary

- **Seam**: A vertical slice of the wargame feature defined by the implementation plan — each seam delivers one self-contained capability. Seam 3 (this document) is the turn cycle engine; Seam 4 is the player room UI; Seam 5 is the GM control UI.
- **GM (Game Master)**: The instructor role that reviews AI-generated draft responses before publishing them to student teams. The GM does not author responses directly — they approve or edit what the AI produces.
- **Turn cycle**: One complete sequence of: timer fires → move buffers locked → AI preprocessing → GM review → publish → next timer scheduled.
- **Move buffer**: A per-team collaborative CRDT document where students draft their orders during the `drafting` phase. Cleared after each publish.
- **Round state**: The state each team is in at any given moment: `drafting` (timer running, students writing), `locked` (deadline passed, AI preprocessing underway or complete, GM reviewing), or `simulation_ended` (terminal state).
- **Bootstrap**: The initial scenario setup for a team — the `scenario_bootstrap` template from `WargameConfig` expanded with the team's codename and stored as the first user message. It seeds the AI's context before any student moves exist.
- **Snapshot pipeline**: The process of extracting plain markdown text from a team's `move_buffer_crdt` at lock time. The snapshot becomes the user message passed to the AI agent.
- **Preprocessing**: The server-side step that runs after `lock_round()` — snapshot extraction followed by a serial AI call per team, producing a draft assistant response and an updated game-state artifact.
- **Game-state artifact**: A GM-only text field (`game_state_text`) tracking the current scenario state. It is injected into each AI call and updated with the AI's returned `TurnResult.game_state`. Students do not see it.
- **Completion gating**: The invariant that `publish_all()` enforces — all teams must have a draft assistant response in the locked state before the round can be published.
- **One-response invariant**: The guarantee that exactly one assistant message exists per team per round. Enforced in `run_preprocessing()` (rejects duplicates) and verified in `publish_all()`.
- **CRDT (Conflict-free Replicated Data Type)**: A data structure that supports concurrent edits from multiple clients without conflicts. Used here via the `pycrdt` library to back the collaborative move buffer and notes editors.
- **pycrdt**: Python bindings for the Yjs CRDT library. Documents contain named subdocuments (`XmlFragment` for ProseMirror content, `Text` for the markdown mirror).
- **Milkdown**: A rich-text editor framework built on ProseMirror. The move buffer uses Milkdown; the CRDT document mirrors editor content as markdown in a `Text("content_markdown")` field for server-side extraction.
- **APScheduler**: Python job scheduling library. Used here with `AsyncIOScheduler` and `SQLAlchemyJobStore` to persist deadline jobs across app restarts. `misfire_grace_time=None` means jobs missed while the app was down always fire on restart.
- **SQLAlchemyJobStore**: An APScheduler job store backend that persists scheduled jobs in the application's PostgreSQL database, ensuring deadline timers survive app restarts.
- **PydanticAI**: A Python library for building AI-powered applications with Pydantic-validated structured outputs. Used here to define `turn_agent` and `summary_agent` with typed return models (`TurnResult`, `StudentSummary`).
- **TestModel**: PydanticAI's built-in deterministic stub model for testing. Allows unit tests to exercise agent logic without making real API calls.
- **Functional core / imperative shell**: An architecture pattern where all logic with no side effects (pure functions) is separated from code that performs I/O (database, network, scheduling). Tests for the pure core require no mocking; tests for the shell mock only the boundaries.
- **`sequence_no`**: The ordering field on `WargameMessage`. Messages within a team are ordered by this integer rather than by timestamp, giving deterministic ordering independent of clock skew.
- **`metadata_json`**: A JSON field on `WargameMessage` used to store machine-readable structured data alongside the human-readable `content`. For wargame messages, it stores PydanticAI's serialised message history on assistant messages, enabling history restoration on the next AI call.
- **Wall-clock mode**: One of two timer configurations on `WargameConfig`. The deadline is a fixed time of day (e.g. 17:00); if that time has already passed when the round is published, the deadline rolls to the next day.
- **Delta mode**: The other timer configuration. The deadline is calculated as `publish_time + timer_delta` (a duration), regardless of time of day.

## Architecture

Functional core / imperative shell, consistent with Seam 2's team management pattern.

**Pure domain logic** (`src/promptgrimoire/wargame/`):
- `turn_cycle.py` — State transition helpers, deadline calculation, CRDT move buffer extraction, prompt assembly, bootstrap template expansion
- `agents.py` — PydanticAI agent definitions (`turn_agent`, `summary_agent`) and structured output types (`TurnResult`, `StudentSummary`)

**Imperative shell** (`src/promptgrimoire/db/wargames.py`):
- Extensions to the existing module: `start_game()`, `lock_round()`, `run_preprocessing()`, `publish_all()`, `on_deadline_fired()`
- Orchestrates DB writes, AI calls, and scheduler interactions

**Scheduler** (`src/promptgrimoire/scheduler.py`):
- APScheduler 3.11.x with `AsyncIOScheduler` and `SQLAlchemyJobStore`
- `init_scheduler()` / `shutdown_scheduler()` called from app lifecycle hooks
- `schedule_deadline()` / `cancel_deadline()` for one-shot date-trigger jobs
- `misfire_grace_time=None` on all deadline jobs — missed deadlines always fire

**Data flow — normal round:**

```
Deadline fires (APScheduler)
  → on_deadline_fired(activity_id)
    → lock_round(): set round_state='locked', clear current_deadline
    → run_preprocessing(): for each team (serial):
        extract_move_text(move_buffer_crdt)        [pure]
        → store WargameMessage(role='user')         [shell]
        → turn_agent.run() → TurnResult             [agents]
        → store WargameMessage(role='assistant')     [shell]
        → update game_state_text                     [shell]

GM publishes (explicit action)
  → publish_all(activity_id)
    → completion gate: verify all teams have draft response
    → for each team:
        summary_agent.run() → StudentSummary         [agents]
        → update student_summary_text                [shell]
    → advance current_round, set round_state='drafting'
    → clear move_buffer_crdt for all teams
    → calculate_deadline() → schedule APScheduler job
```

**Data flow — game start (round 1):**

```
GM starts game
  → start_game(activity_id)
    → for each team:
        expand_bootstrap(template, codename)         [pure]
        → store WargameMessage(role='user', seq=1)   [shell]
        → turn_agent.run() → TurnResult              [agents]
        → store WargameMessage(role='assistant', seq=2) [shell]
        → update game_state_text                     [shell]
    → set round_state='locked', current_round=1
```

The bootstrap follows the same pipeline as any round — the only difference is the "user message" comes from the scenario template rather than a CRDT snapshot. GM reviews and publishes the initial briefing through the normal publish flow.

### PydanticAI Agent Design

Two module-level agents in `wargame/agents.py`:

**Turn agent** — generates draft assistant response and updated game state:
```python
class TurnResult(BaseModel):
    response_text: str   # draft message (includes XML projection tags)
    game_state: str       # updated GM-only artifact

turn_agent = Agent(
    'anthropic:claude-sonnet-4-5',
    output_type=TurnResult,
)
```

**Summary agent** — generates student-safe situation update:
```python
class StudentSummary(BaseModel):
    summary: str          # student-facing text

summary_agent = Agent(
    'anthropic:claude-sonnet-4-5',
    output_type=StudentSummary,
)
```

Each agent has a distinct system prompt:
- `turn_agent` uses `WargameConfig.system_prompt` (immutable, set at game creation)
- `summary_agent` uses `WargameConfig.summary_system_prompt` (new column, configurable per activity — instructs the model to produce only student-safe information)

### Message History Strategy

PydanticAI's serialised message history is stored in `WargameMessage.metadata_json` on the most recent assistant message for each team. On each AI call:

1. Load previous assistant message's `metadata_json` → deserialise PydanticAI history
2. Append the new user message (bootstrap or move snapshot)
3. Call `turn_agent.run(prompt, message_history=restored_history)`
4. Store `result.all_messages_json()` in the new assistant message's `metadata_json`

`WargameMessage.content` is the human-readable copy for display/export. `metadata_json` is the machine-readable copy for model calls. No reconstruction needed — PydanticAI's own serialisation is canonical.

The game-state artifact is NOT part of the message history. It is injected into the user prompt each turn via `build_turn_prompt(move_text, game_state)` and returned updated in `TurnResult.game_state`.

### CRDT Move Buffer Design

Each WargameTeam gets two new bytea columns (Alembic migration):
- `move_buffer_crdt` — collaborative drafting area, cleared on publish
- `notes_crdt` — persistent team notes, never cleared

Each CRDT document contains:
- `XmlFragment("content")` — Milkdown ProseMirror document for the editor
- `Text("content_markdown")` — markdown mirror synced from the editor, used by the snapshot pipeline

This mirrors the annotation page's `response_draft` + `response_draft_markdown` pattern.

**Lock enforcement:** Server-side check in the CRDT persistence path — if `round_state != 'drafting'`, reject the update. UI disables the editor too, but the server is authoritative.

**Lifecycle:**
- `drafting` → move buffer writable by editors
- `locked` → move buffer frozen (server rejects updates)
- On publish → `move_buffer_crdt` set to `None` (fresh buffer for next round)
- Notes → always writable by editors, persists across rounds and states

### Round State Machine

Three states: `drafting`, `locked`, `simulation_ended`.

```
                    start_game()
                         │
                         ▼
               ┌─── locked ◄──── on_deadline_fired()
               │    (preprocessing +         ▲
               │     GM review)              │
               │                             │
               ▼                             │
          publish_all()                      │
               │                             │
               ▼                             │
           drafting ─────────────────────────┘
          (timer ticking,
           students drafting)
```

| From | To | Trigger | Action |
|------|----|---------|--------|
| — | `locked` | `start_game()` | Bootstrap + AI call, no timer yet |
| `locked` | `drafting` | `publish_all()` | Release responses, summaries, schedule deadline |
| `drafting` | `locked` | `on_deadline_fired()` | Lock buffers, snapshot, AI preprocessing |
| any | `simulation_ended` | `end_simulation()` | Cancel timer, read-only (Seam 7) |

All teams transition together. `round_state` is per-team (to support future per-team pacing) but Seam 3 always sets them in lockstep.

### Deadline Calculation

Pure function in `turn_cycle.py`:
- **Delta mode:** `publish_time + config.timer_delta`
- **Wall-clock mode:** Next occurrence of `config.timer_wall_clock` — if already past today, rolls to next day

### Service Layer Contract

Functions added to `db/wargames.py`:

- `start_game(activity_id: UUID) -> None` — expand bootstrap, run initial AI, set locked/round 1. Precondition: teams exist, no messages yet.
- `lock_round(activity_id: UUID) -> None` — set locked, clear deadline. Precondition: all teams drafting.
- `run_preprocessing(activity_id: UUID) -> None` — snapshot + serial AI. Precondition: all teams locked.
- `publish_all(activity_id: UUID) -> None` — summaries, advance round, schedule deadline. Precondition: all teams locked with draft responses.
- `on_deadline_fired(activity_id: UUID) -> None` — idempotent orchestrator: locks teams not yet locked, then preprocesses teams missing draft responses (see Crash Recovery below).

**Completion gating:** `publish_all()` rejects if any team lacks a draft assistant response for the current round.

**One-response invariant:** `run_preprocessing()` checks no assistant message exists for the current round before writing one. `publish_all()` checks exactly one exists.

### Schema Changes (Alembic Migration)

One migration adding:
- `WargameTeam.move_buffer_crdt: bytes | None` (LargeBinary, nullable)
- `WargameTeam.notes_crdt: bytes | None` (LargeBinary, nullable)
- `WargameConfig.summary_system_prompt: str` (Text, NOT NULL, server_default='')

## Existing Patterns

**Functional core / imperative shell** (`wargame/codenames.py`, `wargame/roster.py` → `db/wargames.py`): Seam 2 established this separation. Seam 3 extends it with `turn_cycle.py` (pure) and new functions in `db/wargames.py` (shell).

**CRDT document structure** (`crdt/annotation_doc.py`): XmlFragment + Text mirror pattern used for annotation response drafts. Move buffer follows the same approach.

**CRDT extraction** (`db/crdt_extraction.py`): Pure function that deserialises bytes → applies update → extracts text. Move buffer extraction follows the same pattern but is simpler (single Text field).

**Background task registration** (`__init__.py`): `@app.on_startup` / `@app.on_shutdown` hooks for search worker. APScheduler uses the same lifecycle pattern.

**Message persistence** (`db/models.py:WargameMessage`): `sequence_no` ordering, `metadata_json` for structured data. PydanticAI history stored in `metadata_json` extends this existing field.

**LLM client** (`llm/client.py:ClaudeClient`): Direct Anthropic SDK. PydanticAI is a new pattern that diverges from this — justified because PydanticAI provides structured output validation and is needed for the planned LLM playground feature. The existing `ClaudeClient` remains for roleplay; wargame uses PydanticAI.

## Additional Considerations

**Crash recovery and idempotency:** `on_deadline_fired()` must be idempotent. If the app crashes mid-preprocessing and APScheduler re-fires the job on restart, the function must not corrupt state. Implementation: `lock_round()` skips teams already in `locked` state. `run_preprocessing()` skips teams that already have an assistant message for the current round. This means a partial crash leaves some teams with drafts and some without — the re-fired job completes the remaining teams without duplicating work. The one-response invariant remains intact because preprocessing checks for existing responses before writing.

**PydanticAI history storage:** Storing serialised PydanticAI messages in `metadata_json` couples us to PydanticAI's serialisation format. If PydanticAI changes its format in a future version, a data migration may be needed. The `WargameMessage.content` field remains the human-readable fallback. This trade-off was accepted to avoid reconstruction overhead on every AI call.

**Serial preprocessing:** MVP processes teams serially. For activities with many teams (>10), this could mean significant delay between deadline and GM receiving all drafts. The architecture supports future parallelisation — `run_preprocessing` could dispatch concurrent tasks per team — but this is out of scope.

**PydanticAI Graphs:** Evaluated `pydantic_ai.graph` for modelling the turn cycle as a formal state machine. Deferred as over-engineering — the GM review step is two separate server-side operations with DB state in between, not a suspended workflow. Revisit if the workflow gains complex conditional branching (e.g., per-team approve/reject).
